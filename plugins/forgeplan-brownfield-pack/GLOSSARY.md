# Package Glossary (meta-terms)

> Terms used throughout this package, so the forgeplan agent knows what each word means here.

## Core terms

### Brownfield
An existing project with pre-existing code but without (or with outdated) documentation. Contrast: greenfield (new project, docs written alongside code).

### Factum
(Tier 1) What the code actually does, provable by reading it. Deterministic.

### Intent
(Tier 2) Why the business chose this implementation. Non-deterministic; multiple plausible explanations exist.

### Ubiquitous Language
DDD concept — shared vocabulary between developers, domain experts, and stakeholders. Key deliverable of C1.

### Bounded Context
DDD concept — a specific responsibility boundary with its own model, language, and rules. Our 12 contexts (C1-C12) are bounded contexts for the extraction process itself.

### Aggregate Root
DDD concept — the top-level entity in a cluster; controls invariants for the cluster. Catalogued by `domain-model` artifacts.

### User Journey / Use Case
A named flow from actor trigger to business outcome. May span multiple services / systems.

### Invariant
A statement that must always hold (a business rule). Expressed declaratively, not imperatively.

### Scenario
A Given/When/Then specification of a specific flow. Concrete, often per-use-case.

### Hypothesis
A candidate explanation for an observation, not yet verified. Has lifecycle (`drafted → inferred → verified/refuted/parked`).

### Triangulation
Validating a hypothesis against multiple independent sources (code signals, git history, legacy docs, interviews).

### Confidence
Tag on each intent assertion: `verified`, `strong-inferred`, `inferred`, `speculation`, `unknown`.

### Canonical Document
A standalone document with full content (no references to deleted-code dependencies). Survives code deletion.

### RAG Package
The final output format — chunks + metadata ready for ingestion into a Retrieval-Augmented Generation knowledge base.

### Domain Owner
The human expert who knows the business intent. Authoritative source for `verified` tier.

### Interview Packet
A structured markdown document with clustered unresolved questions, context, and response template. Generated by C7.

### ADI cycle
FPF reasoning cycle: **A**bduction (generate hypotheses) → **D**eduction (predict evidence) → **I**nduction (check).

### F-G-R
FPF scoring: **F**ormality (0-3), **G**ranularity (0-3), **R**eliability (0-3). Used for confidence assessment.

### R_eff
Forgeplan's effective reliability score. Weakest-link aggregation from evidence.

## Tool / system terms

### forgeplan
Artifact-based documentation system. Stores PRDs, RFCs, ADRs, etc. with lifecycle and validation.

### autoresearch
Karpathy-style autonomous iteration engine. Pattern: modify → verify → keep/discard → repeat.

### MCP tool
Model Context Protocol function exposed to AI agents. Forgeplan provides many.

### skill (Claude Code sense)
A discoverable capability with prompt + references. Lives in `.claude/skills/<name>/SKILL.md`.

### slash command (Claude Code sense)
User-invocable command like `/autoresearch:learn`. Lives in `.claude/commands/<name>.md`.

### artifact
A document in forgeplan's workspace with frontmatter + body. Has kind, status, id.

### artifact kind
A category of artifact with specific schema and validation. Current: `prd, epic, spec, rfc, adr, problem, solution, evidence, note, refresh`. New in this package: `glossary, use-case, invariant, scenario, hypothesis, domain-model`.

### projection
The rendered markdown view of an artifact (in forgeplan's `<kind>s/` directory).

### drift
When documentation no longer matches current code (checked by `forgeplan_drift`).

### blind spot
An active decision without linked evidence (checked by `forgeplan_blindspots`).

## Process terms

### Wave
A phase of implementation. We have 5 waves (foundation, use-cases, intent, synthesis, output).

### Pass
Over the extraction process. Pass 1 = factum, Pass 2 = intent generation, Pass 3 = triangulation/synthesis.

### Coverage
Percentage of scope completed (e.g., 85% of entry points mapped to use-cases).

### Entry point
A place where external input enters the system. GraphQL mutation, REST endpoint, queue producer, scheduled job.

### Causal chain
Sequence of actions A → B → C where each step triggers the next.

### Side effect
A change outside the calling function's direct return (DB write, queue publish, event emit, etc.).

### State transition
A change in an entity's state (e.g., order.status changing). Captured as edge in use-case/scenario.

## Semantic terms for the knowledge graph

### Node type
Kind of node in the KG: glossary-term, use-case, invariant, scenario, hypothesis, domain-model, action, entity.

### Relation type
Edge kind in the KG: `defines, triggers, verifies, infers_from, resolved_by, parked_in, catalogs, emitted_by, causes, informs, refines, supersedes, contradicts, based_on`.

### Contradiction
Pair of artifacts making mutually-exclusive claims. Flagged by C9.

## Output terms

### Chunk
A self-contained markdown block suitable for embedding. Typical size: 200-800 tokens.

### Metadata
Frontmatter on a chunk: tags, domain, entity_refs, confidence, artifact_id.

### Neutral JSON
Our output RAG format — not vendor-specific. Converters handle target vector stores.

### Embedding hint
Suggested embedding model or approach noted in metadata, for the RAG host.
