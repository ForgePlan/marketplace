# Process anti-patterns — the workflow footguns

The highest-frequency mistakes — the ones that happen in the gap between "code is written" and "change is shipped". Each is named, diagnosed, fixed.

## P1 — `git add -A` / `git add .` (the blast-radius commit)

**The trap**: staging everything at once to save keystrokes.

**Why it bites**: `-A` sweeps in whatever happens to be in the tree — a `.env`, a generated artifact, a large binary, an unrelated WIP file. The worst cases are silent: a committed secret triggers push-protection (or worse, isn't caught), and a leaked credential cannot be un-leaked by a later commit.

**The fix**: stage specific files only (`git add path/to/file`). This is on the **Forbidden** list in repo CLAUDE.md. Pair it with a one-phrase rationale so the rule survives edge cases (see `claude-md.md` A2).

## P2 — `--no-verify` and force-push (defeating the safety layer)

**The trap**: `git commit --no-verify` to skip a failing hook, or `git push --force` to "fix" a messy branch.

**Why it bites**: the local hook and the branch ruleset exist precisely for the moment you are tempted to skip them. `--no-verify` bypasses the PreToolUse safety gate; `git push --force` can destroy a colleague's commits and is irreversible on a shared branch. The local `safety-hook.sh` blocks force-push and direct pushes to `main`/`dev` — reaching for `--no-verify` is reaching past it.

**The fix**: never `--no-verify`; never `git push --force` (Forbidden). Direct push to `main`/`dev` is forbidden — feature branch → PR → green CI → merge. If a hook fails, fix the cause, do not skip the hook. Admin bypass for a ruleset is `gh pr merge --admin`, not a force-push.

## P3 — vacuous green (trusting a self-report over ground truth)

**The trap**: marking work done because the agent said "done, tests pass" — without looking at the diff.

**Why it bites**: a green suite with an **empty `git diff` is a null result**, not a pass — a suite stays green when nothing changed. Trusting the transcript over the git object store is the PROB-002 failure class: the report is generated by the same process that did the work, so it is not proof.

**The fix**: verify the side-effect against ground truth, never against a self-report. Empty diff on a claimed change = BLOCKER. This is the process-level statement of the agent-layer rule in `agents-and-tools.md` (T1/T2). Reference: repo CLAUDE.md "Ground-truth verification discipline".

## P4 — methodology cocktails (mixing BMAD + SPARC + …)

**The trap**: blending methodologies "to cover all bases" — a bit of BMAD, some SPARC, a dash of Spec Kit on one task.

**Why it bites**: what emerges is **none of them**. The artifacts no longer fit any community pattern, and the team has to re-invent review checklists from scratch — losing the one thing a named methodology buys you (a shared, recognisable shape). Each methodology's discipline is internally coherent; a mix is internally contradictory.

**The fix**: pick **exactly one** methodology per task and commit (the smith router does this — one row, never blended). If the situation genuinely sits between two, surface it for a human decision (`<<NEED_USER_INPUT>>` with ≥3 hypotheses), do not split the difference. Reference: `plugins/fpl-skills/skills/smith/routing-map.md`.

## P5 — not bumping the catalog `metadata.version`

**The trap**: shipping a plugin change by bumping `plugin.json` and the catalog's copy of that plugin's version — but leaving `metadata.version` untouched.

**Why it bites**: `/plugin marketplace update` decides whether to re-pull by comparing `metadata.version`. Bump the other two and skip this one, and the client sees no catalog change and **never fetches the new version**. Silent, no error, "I shipped it but nobody has it" — the most expensive omission in the release flow.

**The fix**: every plugin change bumps **three** numbers — the plugin's `version`, the catalog's copy, and the catalog `metadata.version`. Full rule + worked example in `../plugins/versioning.md`.

## P6 — over-reporting / wall of text

**The trap**: a full structured report for a one-line fix, or a status answer written as a dense unsegmented paragraph.

**Why it bites**: a template on a trivial task is ceremony that buries the answer; a wall of text kills scannability so the reader cannot find the one fact they came for. Both waste the reader's attention — the scarcest resource in a status update.

**The fix**: match format to weight — plain prose for small tasks, structured report only when the work warrants it; break any report into scannable cards, never running prose. The canonical anti-pattern files are `plugins/fpl-skills/skills/forge-report/sections/03-anti-patterns/over-reporting.md` and `.../wall-of-text.md`. (Same spirit as `../claude-md/antipatterns.md` Antipattern 1 — structure over prose.)
