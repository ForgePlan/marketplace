# Онбординг — твой первый час с ForgePlan Marketplace

> Карманный гид. От нуля до автономного диспатча агентов за 30-60 минут.

## Что это

Сейчас ты подключишь к своему Claude Code три штуки: **маркетплейс** (каталог плагинов, что-то вроде package registry для Claude), **smith** (мастер-оркестратор, который читает твой проект и подсказывает, что делать дальше) и **Hindsight** (слой памяти, чтобы агенты не забывали всё после каждой сессии). Вместе они превращают Claude Code из умного ассистента в инструмент, который может вести настоящий инженерный цикл.

Словарь в двух строках:

- **forgeplan** — движок артефактов. Хранит твои PRD, RFC, ADR, evidence и решения как типизированные структурированные записи (а не как разрозненные Markdown-файлы). У каждого артефакта есть статус, связи и оценка качества.
- **smith** — маршрутизирующий агент. Ты говоришь ему «хочу собрать X» или «прод горит», а он подбирает подходящую методологию (из 14 готовых), называет каких специалистов диспатчить и в каком порядке. Сам код он никогда не пишет — он маршрутизирует.

Не пытайся это запомнить — к концу гида smith будет рулить процессом, и ты сам увидишь, что делает каждая часть.

## Что понадобится

- Установленный Claude Code (желательно последней версии)
- Терминал, в котором тебе комфортно
- 30-60 минут концентрации
- (Опционально) сторонний проект как песочница — пустая директория подойдёт

## Шаг 1 — Установка маркетплейса (1 минута)

Внутри Claude Code добавь ForgePlan marketplace как источник каталога, потом поставь два плагина для старта: `fpl-skills` (флагманский набор инструментов workflow) и `agents-pro` (в нём живёт агент `smith`).

```bash
# Добавляем каталог
/plugin marketplace add ForgePlan/marketplace

# Ставим два обязательных
/plugin install fpl-skills@ForgePlan-marketplace
/plugin install agents-pro@ForgePlan-marketplace
```

После каждого `/plugin install` ты увидишь короткое подтверждение вроде `Installed fpl-skills v1.31.1` и подсказку, что появились новые слеш-команды. Набери `/` в чате — увидишь в меню `/smith`, `/forge-cycle`, `/audit` и другие.

> [!TIP]
> Если `/plugin install` говорит «already installed», но новых команд не видно — выполни `/plugin uninstall <имя>` и поставь заново. Это особенность инвалидации кэша в Claude Code; обходной путь работает стабильно. После `/plugin marketplace update` всегда заново запускай `/plugin install`, чтобы подхватить новую версию.

## Шаг 2 — Настройка памяти Hindsight (2 минуты)

Hindsight — это **долговременная память твоих агентов**. Факты, выученные в одной сессии — архитектурные решения, договорённости об именовании, уроки от тяжёлого бага — автосохраняются и подкладываются перед каждым промптом в следующей сессии. Без него агенты каждый раз стартуют с холодной головы.

Работает как маленький локальный сервер в Docker. Внешние API-ключи не нужны — Hindsight использует твою подписку Claude для извлечения фактов.

```bash
# 1. Запускаем сервер Hindsight (разово, работает в фоне)
docker run -d --name hindsight -p 8888:8888 -p 9999:9999 \
  -e HINDSIGHT_API_LLM_PROVIDER=claude-code \
  ghcr.io/vectorize-io/hindsight:latest

# 2. Ставим плагин (подключит авто-хуки + вспомогательные скиллы)
/plugin install fpl-hsmem@ForgePlan-marketplace

# 3. Проверяем здоровье
/fpl-hsmem:status
```

`/fpl-hsmem:status` должен напечатать что-то вроде `bank: <имя-твоего-проекта>`, `memories: 0`, `mental_models: 0`. Чистый пустой банк — именно то, что нужно для нового проекта.

Дальше автоматически происходят три вещи:

- **Перед каждым промптом** — релевантные прошлые знания подтягиваются и молча подмешиваются как контекст (`auto-recall`)
- **После каждого ответа** — разговор сохраняется, если перешагнул порог retain (`auto-retain`)
- **На закрытие сессии** — финальный flush ловит всё, что всплыло в последних поворотах диалога

Думать об этом не надо. Просто работает.

> [!TIP]
> У каждого проекта свой банк памяти, привязанный к текущей рабочей директории. Переключился в другой репо — память автоматически переключилась. Банки никогда не перетекают между проектами.

> [!WARNING]
> Если `/fpl-hsmem:status` падает с «connection refused» — контейнер Docker не запущен. Проверь `docker ps | grep hindsight`. Если контейнера нет, запусти `docker run` из шага выше заново.

## Шаг 3 — Первый проект (5 минут)

Время взять и применить. Возьми директорию — пустую или с кодом — и дай smith поднять forgeplan-workspace.

```bash
# Создаём песочницу, если её нет
mkdir my-app && cd my-app && git init

# Открой Claude Code в этой директории, потом в чате:
/smith-bootstrap
```

Smith проведёт короткое интервью — не больше 5 коротких вопросов. Типичный поток:

1. «Что строим?» — одно предложение
2. «Для кого это?» — целевой пользователь или система
3. «Как выглядит успех?» — измеримый результат
4. «Есть существующий код, на который посмотреть?» — пути или «нет»
5. «Какое-то ограничение или non-goal, который стоит обозначить заранее?» — опционально

Пока ты отвечаешь, smith в фоне диспатчит серию специалистов — `brief-intake` пишет Brief, `goal-planner` черновит PRD, `adr-architect` проверяет, не достаточно ли весомое какое-то решение, чтобы нужен был ADR. В конце ты видишь один структурированный отчёт, а не промежуточный шум диспатча.

Когда закончит, у тебя будет:

- Директория `.forgeplan/` с первым Brief и PRD
- `CLAUDE.md` под этот проект
- Чёткая рекомендация «следующий шаг» (обычно: проверь PRD, потом активируй его)

Это greenfield-путь. Если в директории уже был код, smith направил бы тебя на brownfield-онбординг (Strangler Fig + DDD), но вопросы были бы примерно те же.

> [!IMPORTANT]
> Smith никогда не активирует артефакты и не пишет код сам — он маршрутизирует и рекомендует. Реальную запись делают специализированные агенты под его руководством. Это сделано намеренно: оркестрационная логика отделена от исполнения. Если что-то выглядит не так — можно поднять лог диспатча.

## Шаг 4 — Ежедневный workflow

### Утро

```bash
/smith
```

Smith читает состояние forgeplan — активные артефакты, заблокированные, протухшие черновики — плюс недавнюю активность в git и память Hindsight. Получаешь экран статуса вроде: «3 активных PRD, 1 протухший черновик (PRD-007, не трогали 6 дней), рекомендую продолжить PRD-005 (сейчас на фазе Audit) или начать Brief по фиче биллинга, о которой ты вчера упоминал».

### Когда появилась новая задача

```bash
/smith-plan "<описание задачи>"
```

Два реальных примера:

```bash
# Фича
/smith-plan "добавить обработчик Stripe webhook с retry-with-backoff и идемпотентностью"
# → Строка 3 (новая фича в существующем сервисе). Методология: SPARC + Hexagonal.
#   Диспатч: specification → architecture → coder → tester → code-reviewer → guardian.

# Баг
/smith-plan "юзеры жалуются на нестабильные 504 на эндпоинте /sync, началось вчера"
# → Строка 4 (нетривиальный продовый баг). Методология: RIPER-5 + 5 Whys.
#   Диспатч: debugger → coder → tester → guardian. Evidence: reproducer + root cause.

# Порт дизайн-системы в код
/smith-plan "сгенерировать Pencil дизайн-систему нативно в нашем фреймворке + Storybook"
# → Строка 14 (дизайн-система → код). Методология: CANVAS (hook-gate=Yes — токены до кода).
#   Диспатч: canvas-coordinator → designer → guardian/tester → porter-storybook → coder → porter-framework (опционально, мультифреймворк).
```

Smith выбирает **ровно одну** из 14 методологических строк — он их не смешивает. На выходе — структурированный план, который читается за 30 секунд.

### Когда прод горит

```bash
/smith-plan "API отдаёт 500 на /checkout, клиенты не могут платить"
```

Это попадает в строку 12 (live-инцидент). Получаешь двухфазный ответ:

- **Фаза 1 — стабилизация**: rollback / kill switch / feature flag — то, что быстрее всего остановит кровь. Smith называет, кого диспатчить первым.
- **Фаза 2 — post-mortem**: только после стабилизации запусти безвинный 5 Whys, сделай EVIDENCE-артефакт, реши, нужен ли ADR.

Ты остаёшься в управлении; smith просто убирает паузу «кого диспатчить первым».

### Автономная ночь

```bash
/autorun
```

Запускай это перед сном. Он подберёт незаблокированные артефакты из твоего forgeplan-графа, прогонит каждый через полный пайплайн (Brief → Shape → Build → Audit → Evidence → Activate) и остановится только когда (а) всё закрыто, (б) поймал `sentinel` `NEED_USER_INPUT` или (в) гейт безопасности зафиксировал критическую находку. Утром `/smith` выдаст тебе хендофф-сводку.

> [!WARNING]
> `/autorun` автономный, но в рамках: он не может пушить в `main`, не может деплоить, не может мержить PR. Работает только с артефактами и feature-ветками. Кросс-системные эффекты всё ещё требуют твоей руки.

## Шаг 5 — Справочник команд

Если хочешь сделать X — запускай Y:

| Ситуация | Команда | Что произойдёт |
|---|---|---|
| Только открыл сессию, что делать? | `/smith` | Статус + рекомендация следующего шага |
| Стартую с нуля на новом репо | `/smith-bootstrap` | Greenfield-интервью |
| У меня есть задача в голове | `/smith-plan "<задача>"` | План + последовательность диспатча + методология |
| Сравнить BMAD vs SPARC vs остальное | `/smith-routing` | Образовательная прогулка по 14 строкам |
| Стандартный цикл: фича → активация | `/forge-cycle <PRD-NNN>` | Реактивный 4-layer enforcer до активации |
| Прогнать всё незаблокированное | `/autorun` | Автономный цикл до победы или блока |
| Аудит кода или спринта | `/audit` | Параллельное мульти-экспертное ревью (4-6 агентов) |
| Что меня блокирует? | `/forge-progress` | Прогресс по каждому артефакту в реальном времени |
| Подчистить протухшие черновики | `/forge-cleanup` | Восстановление здоровья, чистка orphan-черновиков |
| Ревью PR | `/code-review` или `/audit <ветка>` | Критическое ревью с находками |
| Проверка покрытия 4-layer пайплайна | `/methodology-check <ID>` | Скор по каждому слою + конкретные пробелы |
| Принять архитектурное решение | `/decision <тема>` | ADR-light или полный, с delta-spec |
| Заменить старый ADR | `/supersede ADR-NNN` | Workflow delta-spec (ADDED/MODIFIED/REMOVED) |
| Отслеживать decay (issues, даты, метрики) | `/decay-watch` | Триггерный сканер отложенных дел |
| Закрытие сессии — сохранить контекст | `/smith handoff` | Сводка для следующего раза |

У каждой команды есть более длинное описание в [USAGE-GUIDE-RU.md](USAGE-GUIDE-RU.md). Заложи этот файл в закладки — это справочник, к которому ты будешь возвращаться.

## Шаг 6 — Где ты нужен (а где нет)

Система спроектирована так, чтобы ты нужен был только в точках принятия решений. Рутинная инженерная работа происходит без тебя. Чёткая граница:

| Ты НУЖЕН | Ты НЕ нужен |
|---|---|
| Brief: что / для кого / критерии успеха | Brief → PRD-конверсия (авто) |
| Тай-брейк методологии (`NEED_USER_INPUT`) | Маршрутизация, когда строка очевидна (авто) |
| Бизнес-приоритеты | Декомпозиция в задачи (авто) |
| Аппрув продового деплоя | Локальные прогоны CI / lint / test (авто) |
| Гейт безопасности на критических находках | OWASP / STRIDE pre-сканы (авто) |
| Мерж PR | Code review + smoke-тесты (авто) |
| Решения по аккаунту / биллингу / вендорам | Установка плагинов / апдейт конфига (авто) |
| Архитектурные one-way doors (необратимые) | Обратимые рефакторинги (авто) |
| Разворот посреди задачи или смена скоупа | Рутинный фикс / фича / рефакторинг (авто) |

Правило большого пальца: smith останавливается и спрашивает тебя только когда (а) маршрут — настоящий тай, (б) это бизнес-решение, (в) сработал гейт безопасности или комплаенса или (г) нужно внешнее действие (мерж / деплой / биллинг). В остальных случаях он работает и докладывает.

## Шаг 7 — Если что-то идёт не так

Короткий список того, на что ты вероятнее всего наткнёшься в первую неделю.

**Команда `forgeplan` не найдена**

Forgeplan CLI отдельный от плагина и едет из другого репо. Поставь его глобально разово и проверь:

```bash
# Установи по forgeplan README; потом:
forgeplan --version
```

Плагин использует MCP-сервер (CLI не нужен для большинства флоу), но несколько скиллов идут через CLI для journal / activity stats.

**`/smith` не отвечает или его нет**

Проверь, что плагин включён:

```bash
/plugin list
```

`agents-pro` и `fpl-skills` оба должны быть `enabled`. Если кого-то нет — `/plugin install <имя>@ForgePlan-marketplace` ещё раз. Если в списке есть, но команды всё равно не появляются — перезапусти Claude Code.

**Hindsight ничего не вспоминает**

```bash
/fpl-hsmem:status
/fpl-hsmem:diagnose
```

`diagnose` пробегает 6-шаговую проверку — сервер доступен, банк существует, хуки прокинуты, opt-out файла нет, MCP-блок в `.mcp.json`, недавняя активность retain. Вывод называет сломанный шаг.

**`/smith-plan` классифицирует не тот контекст**

Уточни описание задачи большими подробностями — что уже есть, какое ограничение, как выглядит «готово». Если две строки реально завязли в ничью, smith выкинет блок `NEED_USER_INPUT` и попросит тебя выбрать. Это by design — коктейли из методологий запрещены.

**Агент говорит «У меня нет прав сделать X»**

Так и должно быть. Оркестратор (Profile B) и большинство ревьюеров `read-only` по дизайну — они аудируют, но не мутируют. Следующий диспатч подберёт Profile A-creator или Profile C-coder, у которого есть права на запись. Не пытайся «фиксить» права; проверь, какого агента ты реально диспатчил.

**CI падает с HTTP 403 / «account suspended»**

Это проблема уровня GitHub-аккаунта, а не кода или пайплайна. Пиши в GitHub support. CI workflow маркетплейса здесь не поможет.

> [!TIP]
> Если застрял — попробуй `/smith` без аргументов. В каком бы состоянии ты ни был, он прочитает ситуацию и предложит конкретный следующий шаг. Это универсальная кнопка «где я».
