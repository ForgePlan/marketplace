# Brownfield-гайд — работа в существующем кодбейзе

> ForgePlan Marketplace в легаси-репах. Когда код писал не ты, а отгружать в нём — теперь твоя задача.

## Когда этот гайд тебе нужен

Читай ЭТОТ гайд (а не [ONBOARDING-RU.md](ONBOARDING-RU.md)), если:

- В репе больше 100 коммитов на момент твоего прихода
- Есть код, который ты не писал, и большую часть авторил кто-то другой
- Есть конвенции, которым нужно следовать, но они нигде не задокументированы
- «Просто переписать» не вариант — есть пользователи, для которых система должна оставаться живой
- Тестовый набор частичный, флакающий, или ты ему до конца не доверяешь
- В графе зависимостей хотя бы одна библиотека отстаёт на major-версию от актуальной

Если репа свежая (`git log` пустой или почти пустой, ты делаешь второй коммит, никто другой ещё не трогал) → сначала читай [ONBOARDING-RU.md](ONBOARDING-RU.md). Greenfield использует другую строку в матрице роутинга smith (строка 1 — BMAD-METHOD + Spec Kit). Brownfield — строка 2 — `Strangler Fig` + `DDD` + `Anti-Corruption Layer`. Менять их местами нельзя.

## Почему brownfield сложнее

Greenfield — это чистый лист. Ты пишешь первую строку, ты задаёшь все конвенции, единственное ограничение — сама задача. Brownfield — противоположность. Ты наследуешь гравитацию легаси: незадокументированные допущения, вшитые в модель данных; конвенции, которые держатся только на code review; зависимости, которые нельзя обновить, потому что три других компонента опираются на их сломанное поведение; и тестовый набор, который рассказывает тебе о том, что *раньше* было правдой, но не о том, что правда сейчас.

Вместе с этим ты наследуешь страх что-то сломать. В greenfield сломать фичу значит отрефакторить свой же вчерашний код. В brownfield сломать фичу значит, что какой-то продакшен-пользователь — которого ты никогда не видел — в три часа ночи обнаружит, что отчёт, который он гоняет каждое утро, теперь возвращает неправильное число. Этот страх рационален, и он меняет арифметику любого твоего изменения.

Этот гайд не пытается этот страх замолчать. Он называет паттерны, которые доказанно работают в условиях brownfield, и говорит, какие из них пропустить, когда ограничения на самом деле не применяются.

## Шаг 0 — Что понадобится

Поставь стандартный набор плагинов ForgePlan Marketplace плюс brownfield-специфичный:

```bash
# Стандартный набор — тот же, что для greenfield-онбординга
/plugin install fpl-skills@ForgePlan-marketplace
/plugin install agents-pro@ForgePlan-marketplace
/plugin install agents-sparc@ForgePlan-marketplace
/plugin install agents-core@ForgePlan-marketplace
/plugin install fpf@ForgePlan-marketplace
/plugin install fpl-hsmem@ForgePlan-marketplace

# Brownfield-специфичный — Discover Agent живёт здесь
/plugin install forgeplan-brownfield-pack@ForgePlan-marketplace
```

> [!TIP]
> Brownfield-пак поставляет канонический `discover` агент — Profile A creator, который гоняет 7-фазный MCP-протокол обнаружения и засевает forgeplan-граф артефактов из существующего кода. Ты будешь использовать его как первый шаг почти в любом brownfield-заходе.

## Шаг 1 — Не bootstrap, а сначала discover

`/smith-bootstrap` — для пустых реп. Он засевает Brief, первый PRD и первый ADR. В brownfield-репе со 100+ коммитов ничего из этого тебе не нужно — не нужно изобретать контекст; код его уже содержит. Тебе нужно его извлечь.

В brownfield ты хочешь:

```bash
/discover
```

Или вызови smith в режиме по умолчанию и пусть он сам тебя смаршрутит:

```bash
/smith
```

В brownfield `/smith` читает `git log`, видит, что история есть, и маршрутит на строку 2 (brownfield-модернизация) [матрицы роутинга](../plugins/fpl-skills/skills/smith/routing-map.md). Первый агент, которого он диспатчит — `discover` из `forgeplan-brownfield-pack`.

> [!WARNING]
> Не запускай `/smith-bootstrap` на brownfield-репе. Он предполагает greenfield-условия и выдаст Brief, который противоречит тому, что код реально делает. Если уже запустил по ошибке — задепрекейть созданные им артефакты (`forgeplan_deprecate <ID>`) до того, как продолжишь. Старые артефакты, противоречащие реальности, отравляют любой последующий автоматизированный обзор.

## Шаг 2 — Что делает Discover Agent (7-фазный MCP-протокол)

Discover Agent гоняет **строгий приоритет источников по тирам**: Tier 1 (код, `git log`, манифесты, схемы) выигрывает у Tier 2 (тесты, JSDoc, CI-конфиги), который выигрывает у Tier 3 (`docs/`, README, wiki). Если доки противоречат коду — побеждает код, а само противоречие становится PROBLEM-артефактом. Это не обсуждается: brownfield-доки обычно протухшие, и доверять им — самый частый способ свалить дискавери набок.

7 фаз идут последовательно (это канон — нельзя перескакивать вперёд на доки):

**Фаза 1 — `detect`.** Определяет язык(и), фреймворк(и), рантаймы, структуру монорепо. Читает `package.json`, `Cargo.toml`, `go.mod`, `pyproject.toml`, `composer.json`, `Gemfile` и т.д. Выход: 1 NOTE-артефакт, классифицирующий тип проекта (Monolith / Microservices / Monorepo / Frontend SPA / Data Pipeline).

**Фаза 2 — `structure`.** Обходит дерево директорий на 3 уровня вглубь, определяет границы модулей и точки входа. Выход: 1 NOTE с деревом модулей, кандидатами в bounded contexts и картой точек входа.

**Фаза 3 — `code`.** Читает точки входа и ключевые модули; картирует публичный API, владеемые типы и схемы БД. Выход: 1 корневой PRD («Project Overview») + 1 RFC на каждый крупный модуль с описанием его публичного API, владеемых типов и зависимостей.

**Фаза 4 — `git`.** Тянет `git shortlog -sn`, последние коммиты и самые изменяемые файлы. Выход: 1 NOTE по контрибьюторам + активность + горячие файлы + 1 PROBLEM-артефакт на каждый выявленный сигнал техдолга (модуль с высокой churn, большие диффы, частые переименования).

**Фаза 5 — `tests`.** Считает тестовые файлы против исходников, определяет фреймворк, сэмплит паттерны покрытия. Выход: 1 EVIDENCE-артефакт с вердиктом (PASS / CONCERNS / BLOCKER) по тестовой базе.

**Фаза 6 — `docs` (всегда последняя в Pass 1).** Читает `docs/`, `README.md`, `CHANGELOG.md`. Тегает каждую находку `[legacy-doc]`, потому что это Tier 3. Кросс-сверяет с находками Фазы 3 по коду — каждое противоречие становится PROBLEM-артефактом со словами «доки говорят X, код делает Y».

**Фаза 7 — `synthesize`.** Гоняет `forgeplan_orphans` + `forgeplan_contradictions`, чтобы вытащить несвязанные или конфликтующие находки. Выход: 1 EVIDENCE-артефакт, обобщающий дискавери — режим, выполненные фазы, топ-5 находок, противоречия, рекомендуемые следующие шаги.

Discover-агент закрывает сессию через `forgeplan_discover_complete`, потом эмитит `<<NEEDS_ACTIVATION: ARTIFACT-ID>>` sentinels на каждый черновой артефакт, чтобы оркестратор их активировал. После этого `forgeplan health` отражает реальную карту репы, а downstream-агенты (`architect-reviewer`, `ddd-domain-expert`, `security-expert`) могут маршрутить от неё.

> [!TIP]
> Для реп >100K LOC запускай `/discover --deep`. Для >2M LOC или критичных для бизнеса — `/discover --full`. Оба добавляют углубляющие проходы, которые фанаут параллельных саб-агентов на каждый модульный RFC. Подробнее по режимам — в [README Discover-агента](../plugins/forgeplan-brownfield-pack/agents/discover/README.md).

## Шаг 3 — Типичные ситуации brownfield и как их разруливать

### Ситуация A: Модернизация монолита

> Пример: Rails 5 монолит с 200 роутами, 80 моделями и моделью `User`, которая разрослась до 47 полей, покрывающих auth, биллинг, профиль и преференсы.

**Паттерн**: `Strangler Fig` + `DDD` bounded contexts + `Anti-Corruption Layer`. Это строка 2 матрицы роутинга smith — каноническая троица brownfield.

**Последовательность**:

```bash
/discover                                              # Pass 1: 7-фазная карта того, что есть
/smith-plan "modernize the User domain into bounded contexts"
# smith выбирает строку 2 → диспатчит ddd-domain-expert (A) → adr-architect (A)
forgeplan_reason PRD-NNN                               # ADI ≥3 гипотезы: lift-and-shift / Strangler / leave-alone
# пишешь ADR по выбранной гипотезе с delta-spec
# реализуешь strangler-вертикаль: новый код читает из легаси через ACL, пишет в новое + легаси
```

Режим провала: попытка сделать все 80 моделей одним PR. Каждая Strangler-вертикаль — один bounded context за раз. Rails-монолит остаётся работать всё это время.

### Ситуация B: Добавить новую фичу в легаси

> Пример: добавить «командные подписки» в 5-летнюю SaaS, которая всегда поддерживала только индивидуальные подписки.

**Паттерн**: `Anti-Corruption Layer` + Hexagonal core для новой фичи. Не загрязняй легаси-модель `Subscription` — оберни её.

**Последовательность**:

```bash
/smith-plan "add team subscriptions alongside individual subscriptions"
# smith выбирает строку 3 (новая фича в существующем сервисе) → SPARC + Hexagonal
# ИЛИ строку 2, если изменение реально требует трогать легаси-границы — smith решает
# specification (A) → architecture (A) → coder (C-coder)
```

ACL — это один класс, `LegacySubscriptionAdapter`, который транслирует между новым агрегатом `Team` и старой строкой `Subscription`. Новый код никогда не видит `Subscription` напрямую.

Режим провала: пропустить ACL «потому что это всего один лишний слой». Через три месяца новый код `Team` забит легаси-формованными заботами, и отрефакторить его дороже, чем было построить.

### Ситуация C: Рефакторинг без поломок

> Пример: разделить 47-полевую модель `User` на `User` (identity) + `Account` (billing) + `Profile` (display) + `Preferences` (config).

**Паттерн**: Branch-by-Abstraction + `Mikado Method`. Старое и новое должны сосуществовать в `main` неделями, пока ты по одному мигрируешь читателей и писателей.

**Последовательность**:

```bash
/smith-plan "split User model into 4 bounded contexts"
# smith выбирает строку 6 (рефакторинг) → Branch-by-Abstraction + Mikado
# architect-reviewer (B, pre-refactor) → adr-architect (A) → coder (C-coder)
# → architect-reviewer (B, post-refactor)
```

Сначала пишешь новые абстракции, шипишь их под feature-флагом, мигрируешь читателей по одному, мигрируешь писателей, потом удаляешь старый код. Mikado трекает дерево зависимостей по мере того, как ты по нему идёшь — каждый раз, когда обнаруживаешь предусловие, записывай его до того, как делать.

Режим провала: пропустить post-refactor `architect-reviewer` EVID. Без него ты сделал код другим, а не лучше, и у BMAD-адверсариального гейта не на что указывать.

### Ситуация D: Обновить major-версию зависимости

> Пример: Rails 5 → 7, React 16 → 18, Node 14 → 20.

**Паттерн**: Branch-by-Abstraction на модуль. Обновляй по одному модулю за раз, под флагом, с регрессионным тестовым стендом, которому ты доверяешь.

**Последовательность**:

```bash
/smith-plan "upgrade React 16 to 18"
# smith выбирает строку 6 (рефакторинг) — major upgrade это рефакторинг с дедлайном
# research-analyst (A) → architect-reviewer (B) → coder (C-coder, на модуль)
# → tester (B, регрессионный набор на модуль)
```

Известные режимы провала: deprecated API, которые компилируются, но ведут себя иначе (например, automatic batching в React 18); каскады peer-зависимостей (major-бамп TypeScript тянет 30 type-пакетов); и library-specific breaking changes, о которых ты не знал, потому что они зарыты в CHANGELOG. Фаза 4 (`git`) Discover-агента часто вытаскивает модули, где это вероятнее всего прячется — те, у которых высокая churn.

### Ситуация E: Security-аудит легаси-кода

> Пример: пре-SOC2 аудит или хардненинг после инцидента.

**Паттерн**: OWASP Top 10 2025 + STRIDE threat modelling, плюс ASTRIDE, если в системе есть AI-компоненты.

**Последовательность**:

```bash
/smith-plan "OWASP audit of the auth + payment paths"
# smith выбирает строку 8 (security audit)
# research-analyst (A) → security-expert (B) → injection-analyst (B)
# → pii-detector (B) → adr-architect (A, для митигаций) → guardian (B-gate)
```

Выход: 3 Profile B EVID (по одному на ревьюера), каждый с `## Findings`, перечисляющим конкретные CWE и шаги ремидиации. Любая митигация, меняющая архитектуру, становится ADR. Тактические митигации (валидация ввода, output encoding) ложатся как fix-PR без ADR.

### Ситуация F: Гасим техдолг

> Пример: 200 lint-варнингов, 1500-строчный god class и три deprecated API, разбросанных по кодбейзу.

**Паттерн**: A3 Problem Solving (одностраничный Toyota-формат) + Fishbone-диаграмма для root cause. Спринты по техдолгу проваливаются, когда команда не может вслух сформулировать, *почему* погасить этот долг сейчас стоит того.

**Последовательность**:

```bash
/smith-plan "tech debt cleanup sprint for Q3"
# smith выбирает строку 11 (техдолг) → A3 + Fishbone
# code-analyzer (C) → research-analyst (A) → architect-reviewer (B)
# → adr-architect (A) → coder (C-coder) → tester (B)
```

A3-лист — Background / Current state / Target state / Analysis / Countermeasures / Plan / Follow-up — помещается на одну страницу. Если не помещается — ты пытаешься чинить слишком много за раз. Выбери одно.

### Ситуация G: Чинить продовый баг

> Пример: race condition в платёжном вебхуке или регрессионный отчёт от клиента.

**Паттерн**: RIPER-5 (Research-Innovate-Plan-Execute-Review) + 5 Whys для root-cause. То же самое, что greenfield-строка 4 — методология не меняется только из-за того, что кодбейз старый.

**Последовательность**:

```bash
/smith-plan "fix the duplicate-charge bug in webhook handler"
# smith выбирает строку 4 (продовый баг)
# debugger (C) → error-detective (C) → research-analyst (A)
# → coder (C-coder) → code-reviewer (B) → tester (B)
```

5 Whys заставляет пройти мимо первой правдоподобной причины. Research-фаза RIPER-5 предотвращает классический паттерн «запатчили симптом, поехали дальше, увидели тот же баг через 6 недель в другой подаче».

## Шаг 4 — `Strangler Fig` + `DDD` + `ACL` — троица brownfield (с конкретными примерами)

Эти три паттерна — главные методологии строки 2 (brownfield-модернизация) у smith. Они работают вместе, и каждого по отдельности недостаточно.

**`Strangler Fig`** (Martin Fowler). Ты не переписываешь. Ты выращиваешь новую систему *вокруг* легаси, пока легаси не съёжится до размера, в котором её безопасно ретайрить. Метафора — лиана-душитель, которая обвивает дерево-хозяина и в итоге его замещает, ни разу его не срубив. В любой момент миграции и старое, и новое работают; в конце старого нет, но big-bang переключения никогда не требовалось. Это risk-averse by design — легаси продолжает приносить выручку всё это время.

**`DDD` bounded contexts** (Eric Evans). Найди швы в легаси, где один «домен» заканчивается и начинается другой. 47-полевая модель `User` обычно содержит 3-5 скрытых bounded contexts, которые исходный автор так и не извлёк. Швы там, где меняется тон названий полей — `email` и `password_hash` (Identity) сидят рядом с `stripe_customer_id` и `billing_address` (Billing), которые сидят рядом с `display_name` и `avatar_url` (Profile). Каждый — свой bounded context со своим ubiquitous language. `Strangler Fig` работает по одному bounded context за раз — `DDD` называет контексты.

**`Anti-Corruption Layer`** (тактический паттерн `DDD`). Трансляционный слой между старыми легаси-концепциями и новыми чистыми доменными концепциями. ACL поглощает всю легаси-странность, чтобы новый код её не видел. Без него новая доменная модель загрязняется за недели — легаси-поле, которым пользуется один потребитель, становится атрибутом твоего чистого entity «просто на время», и это «на время» никогда не заканчивается.

### Конкретный пример — 47-полевой User

Начальное состояние: Rails 5 модель `User` с 47 полями, покрывающими auth-состояние, биллинг-данные, display-профиль и notification-преференсы. У каждой другой модели стоит `belongs_to :user`. Файл модели `User` — 1200 строк.

DDD-анализ выявляет 4 bounded contexts внутри существующей модели:

| Контекст | Поля (сэмпл) | Роль владельца |
|---|---|---|
| **User** (Identity) | `email`, `password_hash`, `mfa_enabled`, `last_login_at` | кто ты есть |
| **Account** (Billing) | `stripe_customer_id`, `plan_tier`, `billing_email`, `seat_count` | как ты платишь |
| **Profile** (Display) | `display_name`, `avatar_url`, `bio`, `timezone` | как тебя видят |
| **Preferences** (Config) | `email_notifications`, `digest_frequency`, `theme` | что ты настроил |

Strangler-вертикаль для `Profile`:

1. Создай новый агрегат `Profile` как отдельную таблицу + Hexagonal-репозиторий
2. Напиши ACL `LegacyProfileAdapter`: читает из `users.display_name` и т.д., пишет и в старую строку User, и в новую строку `profiles`
3. Под feature-флагом маршрутишь новые записи через ACL
4. Бэкфилишь таблицу `profiles` из `users`
5. Мигрируешь читателей по одному потребителю за раз (50+ call-sites — `Mikado Method` их трекает)
6. Когда все читатели мигрированы, перестаёшь dual-writing — только новый путь
7. Удаляешь легаси-поля из `users`

На шаге 7 легаси для `Profile` мертво. Старый `User` ужался на 4 поля. Дальше можно стартовать следующую вертикаль (`Account`, `Preferences` и т.д.). Rails-приложение всё это время работало, и migration window не понадобилось ни разу.

Это и есть троица в действии. `Strangler Fig` — стратегия; `DDD` называет швы; `ACL` держит новый код чистым, пока легаси ещё жив.

## Шаг 5 — Когда НЕ использовать smith-автоматизацию

Smith — не всегда правильный инструмент. Будь честен с собой, в каком ты случае:

- **Крошечные однострочные фиксы.** Опечатка в сообщении об ошибке, пропущенная точка с запятой, ошибка в README. Ручной `git commit` быстрее, чем оркестрировать диспатч. Накладные расходы smith превышают пользу.
- **Исследовательские спайки.** Спайк по определению — эксперимент, который ты можешь выкинуть. Smith хочет выбрать методологию и произвести evidence; спайки пропускают методологию, чтобы быстрее научиться. Используй smith *после* спайка, чтобы выбрать методологию для настоящей работы.
- **Кодбейзы, которые ты выкинешь через 3 месяца.** Миграционный проект, где все знают, что текущий код умрёт через 90 дней. Накладные расходы автоматизации превышают пользу, потому что артефакты тебе уже не пригодятся.
- **Инциденты в 3 часа ночи.** Smith УМЕЕТ обрабатывать incident response (строка 12), но первые 10 минут пожара человек + ранбук быстрее, чем матрица роутинга. Используй smith для пост-инцидентных PRD + ADR, а не для остановки крови.

> [!TIP]
> Эвристика: smith окупается, когда (а) артефакт должен пережить сессии, (б) команде нужно одинаково применять одну и ту же методологию, (в) у работы несколько скоординированных шагов. Иначе — просто делай.

## Шаг 6 — Типичные ловушки и как из них выходить

**Discover-агент произвёл 7 фаз «вот что есть в репе», но что мне с этим ДЕЛАТЬ?** Это вход, а не выход. Discovery — это карта; ситуационные паттерны из Шага 3 — маршруты. Выбери один и вызови `/smith-plan <task>` — артефакты, которые засеял дискавери, дают downstream-агентам (`architect-reviewer`, `ddd-domain-expert`, `security-expert`) контекст, нужный им для реальной работы. Частая ошибка — относиться к самому дискавери как к результату; это не результат, результат — это PR модернизации / ADR / ветка рефакторинга, а дискавери просто делает их возможными без необходимости прочитать всю репу руками.

**ACL раздувается и сам становится легаси.** Пересмотри границу. Иногда ACL делает слишком много, потому что bounded contexts, которые ты нарисовал, неправильные — раздели ACL на два. Иногда ACL раздут, потому что ты пытаешься держать живыми и старое, и новое сильно дольше, чем планировалось завершить миграцию — напиши ADR с жёстким дедлайном и supersede, когда дедлайн придёт.

**`Strangler Fig` застрял посреди миграции.** Какие-то маршруты новые, какие-то старые, обе ветки поддерживаются, и команда про это забыла. Это самый частый режим провала. Forcing function: напиши ADR с дедлайном + supersede, когда последний легаси-маршрут переедет. Без дедлайна состояние «посреди миграции» становится стабильным.

**Brownfield-дискавери произвёл 7 черновых артефактов, которые засоряют `forgeplan_health`.** Либо запусти `/forge-cleanup`, чтобы задепрекейтить те, которые больше не отражают реальность, либо используй `forgeplan_deprecate` напрямую. Не оставляй черновики висеть — они валят гейты валидации и сбивают последующий автоматизированный обзор.

**Smith маршрутит на строку 6 (рефакторинг), а нужна была строка 2 (полная модернизация).** Скажи smith явно, какая строка, или уточни описание задачи. Матрица роутинга детерминистична при ясной задаче; неоднозначные описания дают неоднозначные маршруты. Если две строки реально равны — smith эмитит `<<NEED_USER_INPUT>>` и останавливается.

**Тесты флакают, отсутствуют или врут.** Напиши регрессионный тест ПЕРВЫМ, до любого рефакторинга. Это гейтовый шаг `Mikado Method` — ты не трогаешь имплементацию, пока у тебя нет теста, который падает по правильной причине. Если существующие тесты ненадёжны, цена неправильного рефакторинга неограниченна.

**Обновление зависимости ломает 50 вещей.** Не пытайся починить все 50 одним PR. Branch-by-Abstraction по одному модулю за раз, шипи каждый модуль независимо, чини по одному потребителю за раз. Big-bang upgrade — самый частый способ превратить обновление major-зависимости в 6-месячный проект.

**Bounded contexts, которые я нарисовал, не совпадают с тем, как говорит команда.** Это пробел DDD ubiquitous language, а не brownfield-проблема. Запусти `ddd-domain-expert` по выходу дискавери — он вытаскивает словарь, которым реально пользуется код, против словаря, которым команда пользуется на стендапах. Сам разрыв часто — самая ценная находка всего захода. Перерисуй контексты в терминах команды, а не кода.

**Продакшен-пользователи зависят от легаси-поведения, которое я считаю багом.** Не чини молча. Этот «баг» может быть несущим для внешнего workflow, которого ты никогда не видел. Напиши ADR, документирующий легаси-поведение, явно реши — оставлять, депрекейтить с дедлайном или замещать. `Strangler Fig` даёт тебе взлётную полосу, чтобы сделать это, никого не сломав.
