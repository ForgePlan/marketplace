## 1. Зачем нужна вся эта машина

Когда один человек пишет код для себя - никакая методология не нужна. Когда команда из пяти людей строит продукт на 2 года - нужно нечто, что не даст никому из них (включая помощника-ИИ) тихо срезать угол и потом обнаружить через месяц, что фундамент кривой.

Машина решает три проблемы.

**Первая проблема - уверенность без проверки.** ИИ-помощник пишет уверенно. Артефакты выглядят законченными - пока кто-то третий не прочитает их и не найдёт пробел. Без структурного защитника таких пробелов накапливается +25-41% сложности (по данным MSR 2026). Это не сбой помощника - это свойство порождения текста машиной.

**Вторая проблема - размытие границ ролей.** Один человек в роли архитектора может в любой момент переключиться в роль программиста, и наоборот. Чем длиннее сессия - тем больше переключений, тем труднее проследить, кто и зачем сделал какое решение. В машине роли разделены **физически**: каждая роль - отдельный агент, у каждого свой набор разрешённых инструментов, и переключиться нельзя.

**Третья проблема - потеря контекста между сессиями.** Через неделю никто не помнит, почему выбрали именно эту базу данных. Артефакты (PRD, RFC, ADR, EVID, NOTE) и есть память между сессиями. Машина заставляет писать их прежде чем писать код.

Машина состоит из четырёх сущностей:

- **Артефакты** (PRD, RFC, ADR, EVID, NOTE, EPIC, PROBLEM, SOLUTION, SPEC, REFRESH) - фиксируют решения и проверки
- **Агенты** (пять канонических ролей) - выполняют работу в изолированных контекстах
- **Методологии** (BMAD, FPF ADI, SPARC, RIPER-5 и другие) - определяют как именно подойти к задаче
- **Ворота качества** (forge-cycle, guardian, methodology-check, проверки forgeplan) - не дают пропустить шаги

Дальше - детально про каждую сущность и как они соединяются в один процесс.

## 2. Пять канонических ролей агентов

Каждый агент в системе принадлежит одной из пяти ролей. Роль определяет **что агенту разрешено и что запрещено** на уровне инструментов - через список запретов в его заголовочном блоке (frontmatter `disallowedTools`). Это самая надёжная защита: даже если агент по ошибке захочет сделать что-то не своё, инструмент физически не вызовется.

### Сводная таблица - матрица CRUD-R-A

```
+--------------+-------------------------+----------------------------+-----------------------------+
| Роль         | Что делает              | Что запрещено              | Типичный результат          |
+--------------+-------------------------+----------------------------+-----------------------------+
| Роль A       | Создаёт новые артефакты | Write, Edit, NotebookEdit, | Артефакт в статусе draft    |
| Создатель    | через MCP               | forgeplan_activate         | (PRD, RFC, ADR, NOTE...)    |
+--------------+-------------------------+----------------------------+-----------------------------+
| Роль B       | Проверяет существующий  | Все изменения артефактов,  | EVID с вердиктом            |
| Рецензент    | артефакт враждебно      | activate, reason, claims,  | PASS / CONCERNS / BLOCKER   |
|              |                         | memory_retain              | + ## Findings (≥1 находка)  |
+--------------+-------------------------+----------------------------+-----------------------------+
| Роль C       | Исследует и собирает    | ВСЕ изменения (Write, Edit | Структурированный синтез    |
| Только-чтение| контекст                | Bash, forgeplan_*)         | оркестратору (без файлов)   |
+--------------+-------------------------+----------------------------+-----------------------------+
| Роль         | Пишет исходный код      | Все изменения forgeplan    | Изменения в файлах          |
| C-кодер      |                         | (new, update, link, ...)   | + git коммит в рабочем      |
|              |                         |                            |   дереве                    |
+--------------+-------------------------+----------------------------+-----------------------------+
| Роль D       | Правит существующий     | forgeplan_new, Write,      | Обновлённый артефакт        |
| Сопроводитель| артефакт на месте       | Edit, activate, reason     | (та же ID, изменённое тело) |
+--------------+-------------------------+----------------------------+-----------------------------+
```

Дальше - про каждую роль подробно.

### 2.1 Роль A - Создатель

**Задача**: создать новый артефакт. Одна роль, много специализированных вариантов - каждый под свой тип артефакта.

**Что характеризует**:
- Использует `forgeplan_new` или `forgeplan_generate` через MCP
- Не пишет файлы напрямую (`Write`, `Edit`, `NotebookEdit` запрещены)
- Не активирует свой же артефакт (`forgeplan_activate` запрещён) - это работа оркестратора
- Перед созданием обязан вызвать `forgeplan_reason` - структурированное рассуждение через цикл FPF ADI, выдаёт минимум 3 гипотезы
- Оставляет артефакт в статусе `draft` - дальше его берёт рецензент роли B

**Кто относится** (семь специализированных агентов):

| Агент | Что создаёт | Когда вызывается |
|---|---|---|
| `brief-intake` | Brief NOTE | Самый первый шаг - превращает сырую идею в структурированный артефакт |
| `specification` | PRD или SPEC | После Brief - формирует требования с измеримыми критериями приёмки |
| `architecture` | RFC | После PRD - один RFC на одно PRD |
| `goal-planner` | EPIC + разбиение в несколько RFC | Когда большой PRD надо разбить на параллельные RFC |
| `adr-architect` | ADR | Когда нужно зафиксировать архитектурное решение |
| `artifact-author` | Любой артефакт через `forgeplan_generate` | Универсальный создатель если специализированного нет (PROBLEM, SOLUTION, REFRESH) |
| `discover` (brownfield-pack) | NOTE / ADR из 7-фазной разведки | Знакомство с легаси-кодовой базой через инструменты MCP |

**Жёсткие правила роли A**:

1. Никогда не пишет файлы - только вызовы MCP
2. Перед финализацией артефакта обязательно цикл ADI (минимум 3 гипотезы)
3. Никогда не активирует созданный артефакт - он остаётся в статусе `draft`
4. Каждое утверждение в теле артефакта помечает идентификатором (для следа аудита)

### 2.2 Роль B - Рецензент

**Задача**: прочитать готовый артефакт, найти проблемы, написать отчёт в виде EVID. Сам артефакт не правит - только фиксирует находки.

**Что характеризует**:
- Список запретов включает: `Write`, `Edit`, `NotebookEdit`, `forgeplan_activate`, `forgeplan_reason`, `forgeplan_claims`, `memory_retain`
- Создаёт EVID через `forgeplan_new(kind="evidence", parent_id=...)` - связь типа `informs` к родителю
- В теле EVID обязательный раздел `## Findings` с минимум одной находкой
- Вердикт: PASS / CONCERNS / BLOCKER

**Кто относится** (восемь специализированных агентов):

| Агент | Что проверяет / собирает | Что ищет / делает |
|---|---|---|
| `code-reviewer` | Различия в коде (diff) | Ошибки, нарушения стиля, дублирование, нерабочие крайние случаи |
| `security-expert` | Код или артефакт на безопасность | Категории STRIDE / OWASP Top 10 / CWE |
| `tester` | Набор тестов (запускает через `Bash`) | pass / fail / skipped / нестабильные + изменение покрытия |
| `architect-reviewer` | RFC против родительского PRD | Модульные границы, радиус поражения, пробелы соответствия |
| `system-dev` | RFC против системы целиком | Долгосрочная поддерживаемость, операционные риски на 6+ месяцев |
| `artifact-reviewer` | Любой артефакт на здоровье | Полнота схемы, граф связей, актуальность |
| `evidence-recorder` | Любые внешние результаты (логи, замеры, ручной QA) | Превращает сырой ввод в канонический EVID. Запасной путь когда нет специализированного рецензента |
| `evidence-gatherer` | Активный сбор доказательств для Trust Calculus | Ищет 20-30 источников, оценивает надёжность, синтезирует EVID с F+G+R |

**Жёсткие правила роли B**:

1. Никогда не правит то, что проверяет
2. Пустой `## Findings` - это CONCERNS (повторный запуск)
3. «Никаких проблем» одной строкой - недостаточно враждебно. Нужно объяснение что именно проверял и почему пробелов нет (минимум 2 предложения)
4. Работает в **изолированном контексте** - не видит ни основной переписки, ни обсуждений автора

### 2.3 Роль B-оркестратор (smith) - стратегический планировщик

Особый подвид. Не проверяет конкретный артефакт, не создаёт EVID. Читает **широкое состояние проекта** и возвращает план - какую методологию применять и кого запускать следующим.

**Кто относится**: только `smith` (один на всю экосистему - этого достаточно).

**Что характеризует**:
- Список запретов как у роли B, плюс все изменения forgeplan (`new`, `update`, `link`, ...)
- Не пишет файлы, не запускает агентов сам - возвращает план в виде Markdown
- На вход: снимок состояния проекта (`forgeplan_health`, `git status`, `memory_recall`)
- На выход: план с ровно одним выбранным контекстом из 12 + последовательность запуска агентов

### 2.4 Роль B-ворота (guardian) - финальный мастер

Тоже подвид роли B, но с особой ролью. Не ищет новые проблемы (для этого специалисты), а **взвешивает уже найденные** и выносит бинарный вердикт перед активацией артефакта.

**Кто относится**: только `guardian`.

**Что характеризует**:
- Читает сам артефакт + ВСЮ цепочку EVID (всех предыдущих рецензентов)
- Список запретов запрещает `forgeplan_activate` - guardian не активирует, только пишет инструкцию
- Вердикт: PASS (оркестратор может активировать) / CONCERNS (запустить исправляющего и повторить проверку) / BLOCKER (стоп конвейера)

### 2.5 Роль C - Только-чтение

**Задача**: собрать контекст из внутренних и внешних источников и передать оркестратору. Не сохраняет ничего.

**Кто относится**: `research-analyst`.

**Что характеризует**:
- Список запретов запрещает ВСЕ изменения (включая `Write`, `Edit`, `Bash`, любые `forgeplan_*`)
- Может использовать `WebFetch`, `WebSearch`, `Read`, `Grep`, чтения через MCP
- Возвращает структурированный синтез - оркестратор решает что с этим делать

### 2.6 Роль C-кодер - единственный с правом писать код

**Самая важная роль с точки зрения дисциплины** - это единственный агент во всей экосистеме, которому разрешены `Write`, `Edit`, `Bash` на исходных файлах.

**Кто относится**: только `agents-core:coder`.

**Что характеризует**:
- Список запретов запрещает ВСЕ изменения forgeplan (`new`, `update`, `link`, `activate`, ...)
- Получает на вход уже активированный RFC через MCP
- Работает в `isolation: worktree` - отдельное рабочее дерево git, чтобы параллельные кодеры не наступали друг другу на ноги
- Лимит `maxTurns: 60`
- Когда закончил - пишет «ALL CHECKS PASS» и **передаёт управление обратно оркестратору**, чтобы тот запустил рецензента роли B

**Почему именно один**: разрез между «кто пишет артефакт» и «кто пишет код» - гарантия что код не обгонит документацию. Автор PRD физически не может молча написать код, который не описан в PRD. Кодер физически не может молча создать ADR.

### 2.7 Роль D - Сопроводитель

**Задача**: исправить метаданные или тело уже существующего артефакта **без создания нового**. Например - починить congruence_level, исправить сломанную связь, обновить статус с `draft` на `deprecated`.

**Кто относится**: `artifact-maintainer`.

**Что характеризует**:
- Список запретов запрещает `forgeplan_new` (не создаёт новые), `Write`, `Edit`, `forgeplan_activate`, `forgeplan_reason`
- Использует `forgeplan_update` и `forgeplan_link` на существующих
- Запасной путь последней инстанции - если есть kind-специалист, дёргают его, а не роль D

## 3. Методологии - какая когда применяется

Роль агента говорит **что ему разрешено делать**. Методология говорит **как именно подойти к задаче**. На одну роль может приходиться несколько методологий - например, агент `specification` применяет BMAD для PRD, а `adr-architect` применяет FPF ADI для ADR. Оба - роль A.

### 3.1 BMAD - четыре фазы от идеи до PRD

**Когда применяется**: создание нового PRD. Это методология фазы Specification - превращение сырой идеи в формальный документ требований.

**Четыре фазы**:

```
+------------------+    +------------------+    +------------------+    +------------------+
| B - Brainstorm   |--->| M - Modeling     |--->| A - Architecting |--->| D - Delivery     |
|                  |    |                  |    |                  |    |                  |
| Расширение       |    | Структурирование | | Выбор одного     |    | Финальные        |
| пространства     |    | выбранного       |    | варианта из      |    | артефакты:       |
| решений          |    | направления      |    | прошедших фазу M |    | PRD из 13        |
|                  |    |                  |    |                  |    | разделов         |
| Минимум 3        |    | User stories,    |    | Компромиссы,     |    | + план тестов    |
| гипотезы через   |    | модель данных,   |    | риски, фиксация  |    | + план миграции  |
| FPF ADI          |    | конечный автомат,|    | решения          |    | + чек-лист       |
|                  |    | крайние случаи   |    |                  |    | развёртывания    |
+------------------+    +------------------+    +------------------+    +------------------+
```

**Аналогия**: архитектор жилого дома. Сначала бриф с заказчиком (кто будет жить, бюджет). Потом концепция (этажи, кухня, несущие стены). Потом выбор материалов (железобетон или клеёный брус). Только потом чертежи для рабочих. Пропустишь любую фазу - получишь дом без фундамента или кухню без окна.

**Тринадцать обязательных разделов PRD**:

1. Контекст
2. Проблема
3. Цели (Goals)
4. **Чего не делаем (Non-Goals)** - критично, инженеры рефлекторно пропускают
5. Метрики успеха
6. Целевые пользователи
7. Пользовательские сценарии (User Stories)
8. Критерии приёмки
9. Технические требования
10. Риски
11. Зависимости
12. **Открытые вопросы (Open Questions)** - критично
13. Сроки

Три критичных раздела - **Non-Goals, Open Questions, Risks** - те, без которых проект «забывает» крайние случаи и падает в продакшене.

**Пример Non-Goals** для возможности «Теги в заметках»:

```markdown
## Non-Goals
- НЕ поддерживаем иерархию тегов (родитель / ребёнок) в MVP
- НЕ делаем цвета и иконки для тегов
- НЕ синхронизируем теги между устройствами
- НЕ интегрируемся с внешними системами (Notion, Obsidian)
```

Четыре строки, но именно они спасают от расползания рамок задачи.

### 3.2 FPF ADI - три гипотезы до решения

**Когда применяется**: любое значимое решение. Обязательно - при создании ADR. Также внутри BMAD на фазе Brainstorming.

**FPF** = First Principles Framework (каркас рассуждения от первых принципов), **ADI** = Abduction → Deduction → Induction (цикл рассуждения от Чарльза Пирса).

**Три шага**:

```
+----------------------------------------------------------------------+
| Abduction - порождение гипотез                                       |
|                                                                      |
| Сформулировать минимум 3 гипотезы, каждая претендующая на решение.   |
| Третья обязательно должна быть «do nothing / сжатие рамок».          |
|                                                                      |
| Почему именно 3:                                                     |
|   1 гипотеза  = это не гипотеза, это предопределённый ответ          |
|   2 гипотезы  = ложная дихотомия, обе сформулированы одним автором   |
|   3+ гипотезы = одна может оспорить премиссу двух других             |
+----------------------------------------------------------------------+
                                |
                                v
+----------------------------------------------------------------------+
| Deduction - предсказание следствий                                   |
|                                                                      |
| Для каждой гипотезы - какой будет результат если она верна.          |
| Что измерится, что сломается, что выиграет, какая цена.              |
+----------------------------------------------------------------------+
                                |
                                v
+----------------------------------------------------------------------+
| Induction - проверка доказательствами                                |
|                                                                      |
| Какие у нас есть данные (замеры, документация, исторический опыт)    |
| что подтверждает или опровергает предсказания каждой гипотезы.       |
|                                                                      |
| Если данных нет - запуск evidence-gatherer для сбора.                |
+----------------------------------------------------------------------+
                                |
                                v
                       Выбор одной гипотезы
                       + обоснование почему
                       + фиксация в EVID
```

**Технически** это делается через вызов MCP `forgeplan_reason <ARTIFACT-ID>`. Возвращает структурированный набор гипотез. Агент проверяет, выбирает, фиксирует в новом EVID который связывается с родительским артефактом.

**Жёсткое правило (Sprint Z7, PRD-059)**: для каждого Standard+ артефакта (PRD/RFC/ADR) **обязателен** хотя бы один EVID с минимум 3 разделами `### Hypothesis N` в теле. Без него guardian выдаст BLOCKER.

### 3.3 SPARC - пять фаз для технической реализации

**Когда применяется**: реализация возможности в существующем сервисе. Это методология контекста «новая возможность» (строка 3 матрицы smith).

**Пять фаз**:

```
+------------------------------------------------------------------------+
|                                                                        |
|   S - Specification    Уточнение требований из PRD до уровня,          |
|                        достаточного для реализации                     |
|        |                                                               |
|        v                                                               |
|   P - Pseudocode       Скетчи алгоритмов + анализ сложности            |
|                        (Big-O время + память) ДО написания кода        |
|        |                                                               |
|        v                                                               |
|   A - Architecture     Модули, контракты, поток данных,                |
|                        сигнатуры функций (документ RFC)                |
|        |                                                               |
|        v                                                               |
|   R - Refinement       Цикл TDD red-green-refactor:                    |
|                        падающий тест -> минимальная реализация         |
|                        -> рефакторинг                                  |
|        |                                                               |
|        v                                                               |
|   C - Completion       Полировка: обработка ошибок, крайние случаи,    |
|                        настройка производительности                    |
|                                                                        |
+------------------------------------------------------------------------+
```

Первые три фазы (S+P+A) делает агент роли A `architecture` - на выходе RFC. Последние две (R+C) делает агент роли C-кодер - на выходе код.

**Между фазами** - обязательные ворота качества. После каждой фазы рецензент роли B проверяет результат.

### 3.4 RIPER-5 - пять фаз для прод-ошибки

**Когда применяется**: нетривиальная прод-ошибка (строка 4 матрицы smith). НЕ применяется к тривиальным хотфиксам (опечатка в строке логирования) - для них прямой путь без методологии (строка 5).

**Пять фаз**:

```
+----------------------------------------------------------------------+
|  R - Research        Собрать факты. Логи, трассировки стека, недавние|
|                      коммиты, воспроизводимость.                     |
|                      Агент `error-detective`.                        |
|        |                                                             |
|        v                                                             |
|  I - Innovate        Минимум 3 гипотезы первопричины через 5 Whys.   |
|        |                                                             |
|        v                                                             |
|  P - Plan            Какие проверки сделать чтобы выбрать гипотезу.  |
|        |                                                             |
|        v                                                             |
|  E - Execute         Минимальное исправление + регрессионный тест.   |
|                      Агент `coder` роли C-кодер.                     |
|        |                                                             |
|        v                                                             |
|  R - Review          Разбор без поиска виноватых (отдельный NOTE).   |
+----------------------------------------------------------------------+
```

**5 Whys** - техника внутри фазы Innovate, не отдельная методология. Спрашиваешь «почему?» пять раз, начиная от наблюдаемого симптома, пока не дойдёшь до причины достаточно глубокой, чтобы её можно было устранить навсегда.

**Разбор без поиска виноватых (blameless post-mortem)** - обязательный артефакт после исправления прод-ошибки. Не «кто виноват», а «какое условие позволило этому случиться». Цель - предотвратить класс ошибок, не наказать человека.

### 3.5 Strangler Fig + DDD + ACL - для legacy

**Когда применяется**: модернизация старой системы (строка 2 матрицы smith). Например - переписать монолит на микросервисы, мигрировать с одной БД на другую, заменить устаревший фреймворк.

**Strangler Fig** (по аналогии с фикусом-душителем, который оплетает старое дерево пока то не сгниёт): новый код растёт **вокруг** старого, маршрутизация постепенно переключается, старый код умирает кусок за куском. Никогда не делается переписывание «всё сразу».

**DDD** (Domain-Driven Design, проектирование от предметной области): сначала описывается **ограниченный контекст** (bounded context) - где именно проходит граница нового кода. Без этого Strangler Fig превращается в кашу.

**ACL** (Anti-Corruption Layer, защитная прослойка): прослойка между новым и старым кодом, переводящая модели данных. Чтобы старая «гниль» не проникала в новый код.

**Специальный агент**: `discover` (в плагине `forgeplan-brownfield-pack`) - 7-фазная процедура исследования legacy-кодовой базы через инструменты MCP.

### 3.6 OWASP / STRIDE - для аудита безопасности

**Когда применяется**: аудит безопасности (строка 8 матрицы smith), а также вкраплениями во все остальные строки когда касается чувствительных мест.

**OWASP Top 10** - десять самых частых классов уязвимостей веб-приложений, обновляется каждые 3-4 года.

**STRIDE** - модель угроз от Microsoft:
- **S**poofing - подмена идентичности
- **T**ampering - подмена данных
- **R**epudiation - отрицание действия (нет следа аудита)
- **I**nformation Disclosure - утечка данных
- **D**enial of Service - отказ в обслуживании
- **E**levation of Privilege - повышение привилегий

**Агент**: `security-expert` (роль B). Раскладывает находки по STRIDE / OWASP / CWE.

#### Служебные каталоги (не артефакты в строгом смысле)

В `.forgeplan/` помимо артефактов есть служебные каталоги:

| Каталог | Назначение |
|---|---|
| `claims/` | Захват файлов - какой агент держит захват на какой файл (предотвращение конфликтов в параллельных рабочих деревьях) |
| `discovery/` | Промежуточные результаты 7-фазного агента `discover` |
| `findings/` | Отдельные находки (для интеграции с внешними трекерами) |
| `research/` | Сырые результаты research-analyst до того как они станут NOTE/EVID |
| `lance/` | Внутреннее хранилище forgeplan (векторный индекс, кеш) |

| `trash/` | Мягкое удаление - артефакты удалённые через `forgeplan_delete`, восстанавливаемые через `forgeplan_restore` |

Структура каталогов **зеркалит** kinds API: один файл `.md` на один артефакт, имя файла = canonical ID.

#### Связи между типами артефактов

Артефакты связаны через `forgeplan_link` отношениями:

```
EPIC ---refines---> PRD ---refines---> RFC ---guides---> Код (рабочее дерево)
   |                 |                  |
informs           informs            informs
   |                 |                  |
 EVID              EVID               EVID

PROBLEM ---solves---> SOLUTION ---based_on---> ADR (или PRD)
   |                      |
 EVID                   EVID

ADR (active) ---supersedes---> ADR (новый active)
                                + дельта-описание ADDED/MODIFIED/REMOVED

NOTE (Brief) ---based_on---> PRD
NOTE (исследование) ---informs---> PRD / RFC / ADR
NOTE (разбор) ---informs---> родительское исправление
NOTE (отложенные дела) ---tracks---> открытые задачи
```

Подробнее поток по каждому типу - в следующих разделах.

### 4.1 От сырой идеи до Brief

**Вход**: одна строка от человека. «Хочу добавить теги в заметки». «Нужна авторизация через Google». «Тормозит экспорт PDF, надо ускорить».

**Что происходит**:

```
+------------------------------------------------------+
| Человек: «Хочу теги в заметках»                      |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| Оркестратор: запускает `brief-intake` (роль A)       |
| Передаёт дословно сырую мысль                        |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| brief-intake в своём контексте:                      |
|   1. Структурирует идею (что / зачем / для кого)     |
|   2. Вызывает forgeplan_reason - поднимает скрытые   |
|      допущения автора («теги одного уровня или       |
|      иерархия?», «нужна синхронизация?»)             |
|   3. Создаёт Brief NOTE через forgeplan_new          |
|   4. Возвращает ID артефакта оркестратору            |
+------------------------------------------------------+
                          |
                          v
                  Brief NOTE в статусе draft
                  (ещё не пройден через процесс PRD)
```

**Зачем нужен этот шаг**: чтобы исходная мысль не потерялась. Через 3 недели когда PRD будет переписан 4 раза - Brief остаётся точкой отсчёта.

### 4.2 От Brief до PRD (методология BMAD)

**Вход**: Brief NOTE в статусе `active`.

**Что происходит**:

```
+------------------------------------------------------+
| Оркестратор: запускает `specification` (роль A)      |
| Передаёт: Brief NOTE ID + любой добавочный контекст  |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| specification агент проходит BMAD четыре фазы:       |
|                                                      |
|  --- B - Brainstorming ----------------------------  |
|  forgeplan_reason -> минимум 3 варианта реализации   |
|  (свободные строки / отдельная таблица / иерархия)   |
|                                                      |
|  --- M - Modeling ---------------------------------  |
|  Выбранное направление -> сценарии пользователя,     |
|  модель данных, крайние случаи                       |
|                                                      |
|  --- A - Architecting -----------------------------  |
|  Компромиссы, риски, фиксация решения                |
|                                                      |
|  --- D - Delivery ---------------------------------  |
|  Заполнение 13 обязательных разделов PRD             |
|                                                      |
| Создаёт PRD draft через forgeplan_new                |
| Связывает с Brief NOTE связью `based_on`             |
+------------------------------------------------------+
                          |
                          v
                 PRD draft (статус draft, ещё не активен)
                          |
                          v
+------------------------------------------------------+
| Враждебная проверка (рецензенты роли B)              |
|                                                      |
| Параллельно в изолированных контекстах:              |
|   - artifact-reviewer    - здоровье, схема, связи    |
|   - architect-reviewer   - соответствие, последствия |
|                                                      |
| Каждый создаёт EVID с ## Findings (минимум 1 элемент)|
| Связывает с PRD связью `informs`                     |
+------------------------------------------------------+
                          |
                          v
              Цикл обратной связи (см. ниже Образец A / Образец B)
                          |
                          v
+------------------------------------------------------+
| Guardian (роль B-ворота)                             |
|                                                      |
| Читает PRD + все EVID + оценку R_eff                 |
| Проверяет 4 слоя (S10 ADI / S11 Findings / ...)      |
| Вердикт: PASS / CONCERNS / BLOCKER                   |
+------------------------------------------------------+
                          |
                          v
           Если PASS -> оркестратор -> forgeplan_activate
              Если CONCERNS -> повторный запуск specification
                 Если BLOCKER -> возврат в BMAD
```

**Результат**: активный PRD, готовый к разбиению на RFC.

### 4.3 От PRD до RFC (методология SPARC, фазы S+P+A)

**Вход**: активный PRD.

**Что происходит**:

```
+------------------------------------------------------+
| Оркестратор:                                         |
|   Если один PRD -> один RFC -> запуск `architecture` |
|   Если нужно разбиение -> запуск `goal-planner`      |
|   (раскладывает PRD в DAG из N RFC через GOAP)       |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| architecture агент проходит SPARC первые три фазы:   |
|                                                      |
|  --- S - Specification ----------------------------  |
|  Доп. уточнение требований из PRD                    |
|                                                      |
|  --- P - Pseudocode -------------------------------  |
|  Скетчи алгоритмов + анализ Big-O                    |
|  (делает обычно подагент agents-sparc:pseudocode)    |
|                                                      |
|  --- A - Architecture -----------------------------  |
|  Модули, контракты, поток данных, сигнатуры          |
|                                                      |
| FPF ADI обязателен - минимум 3 архитектурных опции   |
| Создаёт RFC draft через forgeplan_new                |
| Связывает с PRD связью `based_on`                    |
+------------------------------------------------------+
                          |
                          v
                  RFC draft
                          |
                          v
+------------------------------------------------------+
| Враждебная проверка (роль B):                        |
|   - architect-reviewer - соответствие RFC и PRD      |
|   - system-dev         - системный взгляд, радиус    |
|                          поражения на 6+ месяцев     |
|                                                      |
| Каждый создаёт EVID с находками                      |
+------------------------------------------------------+
                          |
                          v
          Цикл обратной связи -> Guardian -> активация
```

**Результат**: активный RFC, готовый к реализации в коде.

### 4.4 От RFC до кода (методология SPARC, фазы R+C)

**Вход**: активный RFC.

**Что происходит**:

```
+------------------------------------------------------+
| Оркестратор: запускает `coder` (роль C-кодер)        |
|                                                      |
| Создаёт изолированное рабочее дерево git             |
| Передаёт: RFC ID + список затронутых файлов          |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| coder в своём рабочем дереве:                        |
|                                                      |
|  --- R - Refinement (TDD red-green-refactor) -----   |
|  1. Пишет падающий тест                              |
|  2. Минимальная реализация чтобы прошёл              |
|  3. Рефакторинг для ясности                          |
|  Повторяет цикл для каждого критерия приёмки         |
|                                                      |
|  --- C - Completion -------------------------------  |
|  Обработка ошибок, крайние случаи, настройка         |
|  производительности                                  |
|                                                      |
| В конце: «ALL CHECKS PASS» + возвращает оркестратору |
+------------------------------------------------------+
                          |
                          v
                  Код в рабочем дереве
                          |
                          v
+------------------------------------------------------+
| Враждебная проверка - три рецензента параллельно:    |
|   - code-reviewer    - ошибки, стиль, дублирование   |
|   - security-expert  - STRIDE/OWASP/CWE              |
|   - tester           - запуск тестов, дельта         |
|                        покрытия                      |
|                                                      |
| Три отдельных EVID                                   |
+------------------------------------------------------+
                          |
                          v
              Цикл обратной связи -> Guardian
                          |
                          v
              PASS -> слияние рабочего дерева -> коммит -> PR
```

**Жёсткое правило**: только `coder` имеет право писать код. Если оркестратор сам через `Write` правит исходник - это **нарушение дисциплины CRUD-R-A** (см. раздел 9).

### 4.5 Архитектурное решение -> ADR (FPF ADI + MADR 3.0)

**Когда нужен ADR**: одно из трёх условий (критерии Sprint Z1):

- Решение затрагивает 3 или более модулей
- Решение **заменяет старое** (supersede)
- Решение **необратимо** (например выбор базы данных, миграция платформы)

**Что происходит**:

```
+------------------------------------------------------+
| Архитектурный вопрос:                                |
| «Какую базу выбрать?» «Как делить сервис?»           |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| Оркестратор: запускает `adr-architect` (роль A)      |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| adr-architect проходит цикл FPF ADI:                 |
|                                                      |
|  Abduction: forgeplan_reason -> минимум 3 опции      |
|             (одна обязательно «do nothing»)          |
|                                                      |
|  Deduction: для каждой - какой исход, цена,          |
|             что измерится, что сломается             |
|                                                      |
|  Induction: замеры, документация,                    |
|             исторический опыт                        |
|                                                      |
| Если 3 или более модулей затронуто:                  |
|   Шаг 5b.1 - запуск `c4-diagram` навыка              |
|   -> создаёт диаграммы C4 L1 + L2 в Mermaid          |
|                                                      |
| Если замена старого ADR:                             |
|   Использует шаблон adr-supersede.md                 |
|   Заполняет 4 раздела дельты:                        |
|     ADDED / MODIFIED / REMOVED / UNCHANGED           |
|   Если REMOVED > 50% - помечается «это не замена,    |
|   а новый артефакт, пиши отдельный ADR»              |
|                                                      |
| Создаёт ADR draft в формате MADR 3.0                 |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| Враждебная проверка:                                 |
|   - artifact-reviewer (всегда)                       |
|   - security-expert (если критично для безопасности) |
|   - architect-reviewer (если затрагивает архитектуру)|
+------------------------------------------------------+
                          |
                          v
              Guardian -> PASS -> активация
                          |
                          v
                Старый ADR -> статус superseded
```

### 4.6 Прод-ошибка -> исправление (методология RIPER-5)

**Вход**: жалоба на прод-ошибку. «У клиента не работает экспорт». «Тормозит API». «Падает в 5% случаев на больших файлах».

**Что происходит**:

```
+------------------------------------------------------+
| Оркестратор классифицирует ошибку:                   |
|   Тривиальная (опечатка) -> строка 5 -> прямое       |
|                              исправление             |
|   Нетривиальная          -> строка 4 -> RIPER-5      |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| R - Research                                         |
| Запуск `error-detective` (роль B): собирает          |
| логи, трассировки стека, недавние коммиты, проверяет |
| воспроизводимость                                    |
| Создаёт EVID с фактами                               |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| I - Innovate                                         |
| Запуск `debugger` (роль B): минимум 3 гипотезы       |
| первопричины через 5 Whys                            |
| Создаёт EVID с гипотезами                            |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| P - Plan                                             |
| Оркестратор: какие проверки сделать чтобы            |
| подтвердить или опровергнуть каждую гипотезу         |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| E - Execute                                          |
| Запуск `coder` (роль C-кодер):                       |
|   - Минимальное исправление                          |
|   - Регрессионный тест (падает без исправления,      |

+------------------------------------------------------+
                          |

+------------------------------------------------------+
| R - Review                                           |
| - code-reviewer + tester по исправлению              |
| - Разбор без поиска виноватых (отдельный NOTE):      |
|   - какое условие позволило этому случиться          |
|   - что изменить чтобы класс ошибок больше не        |
|     возникал (не сам этот баг, а весь класс)         |
+------------------------------------------------------+
                          |
                          v
                  Guardian -> коммит -> PR
```

### 4.7 PROBLEM -> SOLUTION - отдельная пара

**Когда применяется**: когда нужно зафиксировать **проблему отдельно** от её решения. Например: «у нас непредсказуемо растёт задержка хвоста» - это PROBLEM. Решение(я) могут появиться позже, могут быть несколькими альтернативами, могут потребовать отдельных PRD.

PROBLEM **не равен** PRD. PROBLEM описывает что плохо. PRD описывает что построим. Разделение нужно когда:

- Решение не очевидно и нужно сначала формализовать проблему
- Одна проблема порождает несколько потенциальных решений (несколько кандидатов SOLUTION)
- Проблема обнаружена, но решать её сейчас нет ресурсов (PROBLEM активна, SOLUTION пока нет)

**Что происходит**:

```
+------------------------------------------------------+
| Наблюдаемая проблема                                 |
| («плагины пользователей не обновляются»,             |
|  «у клиентов теряются заметки при offline»)          |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| Оркестратор: запускает `artifact-author` (роль A)    |
| с kind="problem"                                     |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| artifact-author создаёт PROBLEM-NNN:                 |
|   - Симптомы (что наблюдается)                       |
|   - Контекст (когда случается, у кого)               |
|   - Воспроизводимость                                |
|   - Что НЕ известно (открытые вопросы)               |
|   - Связанные артефакты                              |
| Через forgeplan_new(kind="problem")                  |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| Враждебная проверка через artifact-reviewer (роль B) |
| Ищет: воспроизводимость доказана? Симптомы отделены  |
| от причин? Открытые вопросы перечислены?             |
+------------------------------------------------------+
                          |
                          v
                Guardian -> активация PROBLEM
                          |
                          v
   Дальше - отдельным циклом - рождаются SOLUTION-ы:
                          |
                          v
+------------------------------------------------------+
| Для каждой SOLUTION:                                 |
| Оркестратор -> artifact-author с kind="solution"     |
|   - Один из подходов решения PROBLEM-NNN             |
|   - Компромиссы, цена, риски                         |
|   - Связывается с PROBLEM связью `solves`            |
+------------------------------------------------------+
                          |
                          v
              Проверка + Guardian -> активация SOLUTION
                          |
                          v
       Выбранная SOLUTION -> связь с PRD типа `based_on`
                          |
                          v
            Дальше - стандартный поток PRD -> RFC -> Код
```

**Замечание**: SOLUTION не равен ADR. SOLUTION - это «вот один из вариантов решения». ADR - это «вот окончательное архитектурное решение». Часто PROBLEM порождает несколько кандидатов SOLUTION, обсуждение между ними финализируется в ADR.

### 4.8 EPIC - разбиение многоспринтовой инициативы

**Когда применяется**: когда работа явно не помещается в один PRD. Например: «миграция с MongoDB на PostgreSQL», «выпуск мобильного приложения», «4-Layer Pipeline Enforcement» (как наш EPIC-001).

**Что происходит**:

```
+------------------------------------------------------+
| Многоспринтовая инициатива                           |
| («перейти с MongoDB на PostgreSQL»)                  |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| Оркестратор: запускает `goal-planner` (роль A)       |
| с большим контекстом + указанием kind="epic"         |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| goal-planner применяет GOAP                          |
| (Goal-Oriented Action Planning + поиск A*):          |
|                                                      |
|  1. Определяет конечное состояние                    |
|     «все клиенты переехали, MongoDB выведен»         |
|                                                      |
|  2. Строит DAG из промежуточных состояний            |
|     («схема создана» -> «миграция написана» ->       |
|      «новые клиенты пишут в PG» -> «старые           |
|      перенесены» -> «MongoDB только-чтение» ->       |
|      «MongoDB удалён»)                               |
|                                                      |
|  3. Каждое промежуточное состояние = отдельный PRD   |
|                                                      |
|  4. Создаёт EPIC через forgeplan_new(kind="epic")    |
|                                                      |
|  5. Создаёт N PRD draft (по одному на шаг DAG)       |
|     Связывает каждый с EPIC связью `refines`         |
|                                                      |
|  6. Помечает зависимости (PRD-2 blocked_by PRD-1)    |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| Враждебная проверка EPIC через architect-reviewer    |
| Ищет: разбиение полное? DAG без циклов?              |
| Параллельные пути использованы где можно?            |
+------------------------------------------------------+
                          |
                          v
              Guardian -> активация EPIC
                          |
                          v
       Каждое PRD проходит свой стандартный поток:
       BMAD -> проверка -> Guardian -> активация -> RFC -> Код
                          |
                          v
       EPIC закрывается когда **все** дочерние PRD активны
       и доведены до продакшена
```

**Жизненный цикл EPIC**:

- `draft` - DAG нарисован, дочерние PRD ещё не активны
- `active` - хотя бы один дочерний PRD активирован, работа идёт
- `closed` - все дочерние PRD завершены, EPIC выполнен

**EPIC в нашем проекте** (на 2026-05-28): три активных:

- EPIC-001 - 4-Layer Pipeline Enforcement (Sprints Z6-Z10)
- EPIC-002 - smith master-orchestrator
- EPIC-003 - Sprint AA: methodology auto-enforcement gates

### 4.9 SPEC - узкая техническая спецификация

**Когда применяется**: для **узких технических артефактов** - формат файла, протокол сообщений, схема API, набор правил линтера. SPEC не равен PRD: PRD продуктовый, SPEC технический.

Примеры реальных SPEC:

- Формат канонического тела EVID (markdown bold-pattern)
- Схема `session-checkpoint.yaml`
- Протокол маркера `<<NEED_USER_INPUT>>`
- Структура заголовочного блока агента

**Что происходит**:

```
+------------------------------------------------------+
| Техническая необходимость:                           |
| «нужен стандартный формат для X»                     |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| Оркестратор: запускает `specification` (роль A)      |
| с kind="spec" - или `artifact-author` как запасной   |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| specification создаёт SPEC-NNN:                      |
|   - Цель спецификации                                |
|   - Точный формат (поля, типы, обязательность)       |
|   - Примеры валидные и невалидные                    |
|   - Версионирование                                  |
|   - Совместимость с предыдущими версиями             |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| Враждебная проверка через artifact-reviewer (роль B) |
| Ищет: крайние случаи покрыты? Версионирование явное? |
| Примеры исчерпывающие?                               |
+------------------------------------------------------+
                          |
                          v
              Guardian -> активация SPEC
                          |
                          v
         SPEC становится контрактом - все артефакты
         и код должны следовать ему
```

### 4.10 NOTE - заметки разных подтипов

NOTE - самый универсальный kind. Используется когда нужно зафиксировать что-то структурированно, но это не тянет на PRD/RFC/ADR. По назначению различают подтипы:

| Подтип NOTE | Кто создаёт | Когда | Пример |
|---|---|---|---|
| **Brief** | `brief-intake` | Вход в конвейер - сырая идея -> структура | NOTE-001..018 в инбоксе |
| **Результаты исследования** | `research-analyst` (через оркестратор) | После широкого исследования | NOTE-004 «Comparable systems research» |
| **Трекер отложенных дел** | оркестратор (одна на весь проект) | Каждое отложенное записывается строкой | NOTE-013 - центральный каталог отложенного |
| **Разбор без поиска виноватых** | оркестратор после исправления ошибки | После закрытия RIPER-5 | «какое условие позволило этому случиться» |
| **Находки разведки** | агент `discover` (legacy) | Каждая фаза 7-фазной разведки | NOTE-014..017 «discover-smoke-...» |
| **Сводка закрытия спринта** | оркестратор в конце спринта | После Sprint A-E, J-K, U-V | NOTE-007 «PRD-026 implementation closure» |

**Жизненный цикл NOTE**:

- Может быть `active` (полезная информация) или `deprecated` (информация устарела)
- Может быть **сиротой** (без связей) - это сигнал для `/decay-watch`
- Может быть **залежавшимся черновиком** (29+ часов в `draft` без связей) - тоже сигнал

### 4.11 REFRESH - обновление устаревшего знания

**Когда применяется**: когда у активного артефакта (PRD/RFC/ADR/SPEC) обнаружено небольшое расхождение с реальностью, и нужно **обновить часть информации** без полной замены.

REFRESH - компромисс между «всё в порядке» и «нужен новый ADR с дельтой»:

- Если расхождение системное и решение пересматривается -> замена через новый ADR
- Если расхождение фактическое (цифры, ссылки, пути) -> REFRESH

**Что происходит**:

```
+------------------------------------------------------+
| Обнаружено: «в PRD-049 написано что обработчик в     |
| /a/b/c, но мы давно переехали в /x/y/z»              |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| Оркестратор: запускает `artifact-author` (роль A)    |
| ИЛИ `artifact-maintainer` (роль D - если правка      |
| на месте без нового артефакта)                       |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| Создаёт REFRESH-NNN или правит существующий          |
| Связывает с родительским артефактом связью           |
| `refreshes`. Описывает: что обновлено, источник      |
| правки, когда обнаружено                             |
+------------------------------------------------------+
                          |
                          v
                  Проверка -> активация
```

REFRESH - лёгкий механизм. Не требует цикла FPF ADI (это не решение). Требует проверки только если правка существенная.

## 5. Защитник качества - враждебная проверка

### 5.1 Принцип чужих глаз

Автор не видит свои опечатки. Это **физиологическое свойство восприятия**, не слабость воли. То же относится к ИИ-помощникам: модель защищает свой предыдущий ответ - не из злого умысла, а из устройства внимания, свежий текст «убедительнее» абстрактного правила «ищи проблемы».

Решение - **враждебная проверка в новом контексте**. Открываешь чистую сессию, даёшь роль критика с явной установкой: найти минимум 3 проблемы, не хвалить.

В нашей машине это **программное правило**: каждый рецензент роли B запускается через инструмент `Task`, что создаёт **новый изолированный контекст подагента**. Он не видит основную переписку, не видит обсуждений автора, не видит почему было принято то или иное решение. Только сам артефакт через `forgeplan_get`.

### 5.2 Изоляция контекста как жёсткое правило

Каждый вызов через `Task` = **новая чистая сессия** на уровне модели:

- Не передаётся вся основная переписка - только сформированный оркестратором запрос
- Подгружается заголовочный блок подагента (его собственные инструкции, список запретов)
- После завершения возвращается только финальный результат

Это решает ровно ту проблему, про которую говорит BMAD: «открой новую пустую сессию». Только программно - невозможно случайно «подсмотреть» предыдущий контекст.

### 5.3 Обязательные разделы отчёта

**Sprint Z6 BMAD discipline** (PRD-057) устанавливает жёсткие требования к каждому EVID роли B:

1. **Минимум 1 EVID роли B обязателен** для каждого Standard+ PRD/RFC/ADR перед активацией. Ноль проверок = BLOCKER на воротах guardian.

2. **Тело EVID должно содержать раздел `## Findings` с минимум одной находкой**. Ноль находок = рецензент был недостаточно враждебен.

3. **Что писать когда реально нет проблем**: одной строки «no findings» **недостаточно** - это читается идентично «рецензент не смотрел». Нужно минимум 2 предложения объясняющих **что именно проверял** и **почему пробелов нет**. По умолчанию ожидание: минимум 1 находка существует. Реально безгрешные артефакты - исключение.

4. **Пример хорошей находки**:

```markdown
## Findings

1. **[Severity: HIGH]** AC-3 не имеет измеримого порога - «система должна
   отвечать быстро» не измеримо.
   Рекомендация: заменить на «Система отвечает за 200ms на p95 при 1000
   одновременных пользователях».

2. **[Severity: MEDIUM]** В Non-Goals не указано что не поддерживается
   синхронизация между устройствами - велик риск что в Sprint 2 это
   возникнет как «само собой разумеющееся».

3. **[Severity: LOW]** Раздел Risks не упоминает что миграция данных
   потребует простоя - это надо явно зафиксировать.
```

### 5.4 Два цикла обратной связи

Когда рецензент нашёл проблемы - есть два образца реакции. Выбор зависит от **тяжести и количества находок**.

#### Образец A - быстрое исправление (короткий цикл)

Применяется для находок тяжести LOW / MEDIUM от одного рецензента:

```
Роль A создаёт draft
                |
                v
Роль B рецензент -> EVID с находками (MEDIUM)
                |
                v
Оркестратор -> повторный запуск роли A с конкретными правками
                |
                v
Роль A -> исправление -> новый draft
                |
                v
Роль B рецензент (новый контекст) -> EVID #2:
            «находки из EVID #1 устранены, новых нет»
                |
                v
Guardian -> PASS -> активация
```

Быстро (2-3 хода в диалоге), не накапливает долг, автор сразу знает в чём ошибся.

#### Образец B - ворота с переделкой (длинный цикл)

Применяется для находок HIGH / CRITICAL или когда несколько рецензентов нашли одно и то же:

```
Роль A создаёт draft
                |
                v
Несколько рецензентов роли B (параллельно) -> EVID #1, #2, #3
                |
                v
Guardian читает ВСЁ -> CONCERNS / BLOCKER
                |
                v
Если CONCERNS: оркестратор -> запускает исправляющего (часто тот же
            роли A с другими инструкциями) -> новый draft ->
            повтор проверки
                |
                v
Если BLOCKER: конвейер останавливается, требуется решение человека.
            Возможно - переделка с другой методологией
            (например - пересмотр строки в матрице smith)
```

Медленнее, но даёт **стратегический взгляд**: «три разных рецензента нашли одну проблему - это не баг в PRD, это баг в подходе».

#### Правило выбора

```
+--------------------+------------------------+---------------------+
| Тяжесть находок    | Кто решает             | Образец             |
+--------------------+------------------------+---------------------+
| LOW (мелочи)       | Оркестратор сам        | A - быстрое исправ. |
| MEDIUM (одна, от   | Оркестратор сам        | A - быстрое исправ. |
| одного рецензента) |                        |                     |
| HIGH (любая)       | Guardian               | B - через ворота    |
| CRITICAL           | Guardian + возможно    | B - обязательно     |
| или несколько      | человек через          | через ворота        |
| рецензентов        | <<NEED_USER_INPUT>>    |                     |
| согласны           |                        |                     |
+--------------------+------------------------+---------------------+
```

Эта логика закодирована как **разрешение на 3 уровнях** (AUTO / ADI / USER) в ментальной модели `mm-pipeline-anomalies`.

## 6. Guardian - финальные ворота мастера

Guardian - это **роль арбитра**, последний шаг перед активацией любого артефакта. Не ищет новые проблемы, а **взвешивает уже найденные**.

### Что делает Guardian

```
+------------------------------------------------------+
| Вход: ID артефакта готового к активации              |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| Шаг 1. Чтение через MCP                              |
|   - forgeplan_get(artifact_id) - сам артефакт        |
|   - граф forgeplan_link - все связи и EVID           |
|   - forgeplan_score - текущий R_eff                  |
|   - forgeplan_health - окружающее состояние          |
|   - forgeplan_phase - позиция в жизненном цикле      |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| Шаг 2. Сверка с чек-листом 4 слоёв:                  |
|                                                      |
|   S10 FPF:    Есть EVID с минимум 3 разделами        |
|               ### Hypothesis ?                       |
|                                                      |
|   S11 BMAD:   Есть EVID роли B с непустым            |
|               ## Findings (минимум 1 элемент) ?      |
|                                                      |
|   S12 OpenSpec: Для замены старого есть дельта       |
|                 (ADDED/MODIFIED/REMOVED/UNCHANGED) ? |
|                 Для 3+ модулей есть диаграммы C4 ?   |
|                                                      |
|   S13 Forgeplan: Правильные связи parent/child ?     |
+------------------------------------------------------+
                          |
                          v
+------------------------------------------------------+
| Шаг 3. Вердикт (бинарный):                           |
|                                                      |
|   PASS      -> оркестратор может вызвать activate    |
|   CONCERNS  -> запуск исправляющего + повтор проверки|
|   BLOCKER   -> стоп конвейера, артефакт в draft      |
+------------------------------------------------------+
```

### Важные нюансы

**Guardian не активирует**. `forgeplan_activate` в его списке запретов. Guardian пишет инструкцию (это и есть его EVID), оркестратор её читает и решает следующий шаг.

**Guardian работает в изолированном контексте** - не видел обсуждений, не видел как автор объяснял свои решения. Только артефакт и цепочка EVID.

**Guardian - последний барьер**. Если артефакт прошёл Guardian, считается что он готов к использованию остальной системой. Это значит - Guardian берёт на себя ответственность за качество. Поэтому если CONCERNS - конвейер обязан подчиниться, нельзя «обойти».

### Когда Guardian неприменим

Для **тривиальных задач** (опечатки, переименование переменной, обновление README) Guardian не запускается. Тривиальные задачи идут по строке 5 матрицы smith - прямое исправление без формальной методологии. Запускать Guardian на каждую опечатку - это процессная нагрузка превышающая стоимость исправления.

## 7. Проверки через forgeplan

Помимо человеческих и агентных проверок есть три **автоматических** механизма от системы forgeplan. Они работают постоянно, не только на воротах.

### 7.1 R_eff - автоматическая оценка

**Что это**: число от 0 до 1, считающее «эффективную надёжность» артефакта.

**Формула**: F + G + R

- **Foundation** (основание) - насколько прочны источники (родительские артефакты, доказательства)
- **Granularity** (детальность) - детальность (число разделов, длина тела, явные данные)
- **Reliability** (надёжность) - надёжность каждого доказательства по явной таблице

**Особенность**: считается **без вызова ИИ**, чисто по структуре графа артефактов. Дёшево, повторяемо, объективно.

**Команда**: `forgeplan_score <ARTIFACT-ID>`

**Где использовать**: смотреть R_eff главных PRD периодически. Если у топ-PRD низкий R_eff - это сигнал что либо мало доказательств, либо те что есть слабые.

### 7.2 Правила FPF - фоновая проверка

**Что это**: 5 правил которые проходят по всему графу артефактов и помечают подозрительные.

**Пять правил по умолчанию**:

```
+------------------+----------------------------------------------------+
| Правило          | Что ловит                                          |
+------------------+----------------------------------------------------+
| blind-spot       | Артефакт активен, но критический раздел пустой     |
| weak-evidence    | Есть EVID, но R_eff низкий                         |
| orphan-active    | Артефакт активен, но без связей (сирота)           |
| medium-quality   | Формальные критерии не выдержаны                   |
| ready-to-build   | Все зависимости активны, можно стартовать          |
+------------------+----------------------------------------------------+
```

**Команда**: `forgeplan_fpf_rules`

**Где использовать**: запускается автоматически в `forgeplan_health`. Нарушения выводятся в `next_actions`.

### 7.3 `/methodology-check` - покрытие по четырём слоям

**Что это**: навык только для чтения который для одного артефакта показывает покрытие по всем четырём слоям конвейера (S10-S13) + C4 (если применимо).

**Возвращает**:

- Оценку по слою: 0 (нет), 1 (частично), 2 (полностью)
- Общий процент
- Конкретные дела по каждому пробелу

**Команда**: `/methodology-check <ARTIFACT-ID>`

**Когда вызывать**: **перед каждой активацией** Standard+ артефакта как финальная проверка вменяемости.

**Не правит автоматически** - только показывает что нужно сделать. Это сознательное решение: автоматическая правка может молча сломать намерение автора.

**Пример вывода**:

```
Methodology coverage for PRD-049
================================
S10 FPF ADI         : FULL    (2/2) - EVID-072 has 3 hypotheses
S11 BMAD review     : PARTIAL (1/2) - Findings present but only 1 item,
                                      Sprint Z6 expects >=2 sentences of
                                      justification on zero-gap claim
S12 OpenSpec        : MISSING (0/2) - no delta-spec for supersede
                                      operation linking to ADR-003
S13 Forgeplan       : FULL    (2/2)
C4 (>=3 modules)    : N/A - only 2 modules touched

Aggregate: 6/8 = 75% - ACTIVATION NOT RECOMMENDED

Action items:
1. Re-dispatch artifact-reviewer with mandate to expand Findings
2. Run /supersede ADR-003 to generate delta-spec
```

## 8. Четырёхслойный конвейер S10-S13

Это **обязательный фильтр** через который проходит каждый Standard+ артефакт перед активацией. Расшифровка:

```
+------------------------------------------------------------------------+
|                                                                        |
|  S10  FPF       Проектирование - минимум 3 гипотезы через ADI          |
|  ---  ---       Принуждается: /forge-cycle Шаг 4.5 + guardian Шаг 5    |
|                 Без EVID с 3 разделами ### Hypothesis = BLOCKER        |
|                                                                        |
|         |                                                              |
|         v                                                              |
|                                                                        |
|  S11  BMAD      Ворота качества - проверка роли B минимум 1 находка    |
|  ---  ----      Принуждается: /forge-cycle Шаг 6.5 + guardian Шаг 5    |
|                 Без EVID с непустым ## Findings = BLOCKER              |
|                                                                        |
|         |                                                              |
|         v                                                              |
|                                                                        |
|  S12  OpenSpec  Структура - связи DAG + дельта при замене              |
|  ---  --------  Принуждается: шаблон adr-supersede + /supersede +      |
|                 /decay-watch Шаг 2e                                    |
|                 Без дельты или без правильных связей = CONCERNS        |
|                                                                        |
|         |                                                              |
|         v                                                              |
|                                                                        |
|  S13  Forgeplan Автоматизация - обработчики + агенты + навыки + MCP    |
|  ---  --------- Принуждается: validate + score + ворота активации      |
|                 Без правильной обвязки = технический долг              |
|                                                                        |
+------------------------------------------------------------------------+
```

**Плюс C4** (ортогональный слой) - авто-рекомендуется для решений касающихся 3+ модулей через `adr-architect` Шаг 5b.1. Это не часть основного четырёхслойного конвейера, а дополнение.

**Мотивация**: исследование MSR 2026 показало что ИИ-помощь **без структурных контролей** даёт +25-41% сложности кода. Каждый из четырёх слоёв закрывает одну из категорий проблем:

| Слой | Какую проблему ИИ закрывает |
|---|---|
| S10 FPF | Уверенность без альтернатив - модель уверенно предлагает первое решение |
| S11 BMAD | Уверенная неполнота - артефакт выглядит готовым с пропущенным критичным |
| S12 OpenSpec | Потеря контекста при замене - «забываем» что и почему изменили |
| S13 Forgeplan | Несинхронизированные обвязки - код есть, но обработчики/тесты/CI не подтянуты |

**Команда для проверки**: `/methodology-check <ID>` (см. раздел 7.3).

## 9. Дисциплина усиления - четыре правила

Эти четыре правила - практический минимум для удержания дисциплины. Не методология, а **процессная гигиена**.

### Правило 1. `/methodology-check` перед каждой активацией

Прежде чем вызывать `forgeplan_activate` на Standard+ артефакте - запускай `/methodology-check <ID>` и убедись что покрытие 100% по всем применимым слоям.

**Почему**: это финальная проверка только для чтения. Найдёт пробелы которые ты или агенты могли пропустить (например - забыли дельту при замене). Стоит 5 секунд, экономит часы переделок.

### Правило 2. `/forge-cycle` как точка входа для Standard+

Для любой задачи масштаба Standard и выше - заходи через `/forge-cycle`. Эта команда **принуждает** ключевые шаги:

- Шаг 4.5 - FPF ADI обязателен
- Шаг 6.5 - враждебная проверка BMAD обязательна

Если делать руками - легко срезать угол. Через `/forge-cycle` срезать угол нельзя.

**Когда НЕ применять**: тривиальные исправления (строка 5 матрицы). Опечатка в README не должна проходить через девятиэтапный конвейер.

### Правило 3. Для критичных решений - два разных рецензента

На критичный артефакт (например - PRD платформенного компонента, ADR выбора базы данных) запускай **двух разных рецензентов роли B** параллельно. Например - `artifact-reviewer` + `architect-reviewer` на PRD.

**Зачем**: если оба независимо нашли одну и ту же проблему - это **сигнал высокой надёжности**. Если только один нашёл - может быть его частное мнение. Дешёвый способ повысить чувствительность.

### Правило 4. Доверяй образцу A (быстрое исправление)

Большинство находок - LOW/MEDIUM. Для них **дешевле** сделать повторный запуск роли A с конкретными правками, чем гонять через Guardian. Ворота Guardian - для серьёзных случаев.

**Признак что нужен образец B**: либо одна находка HIGH или выше, либо несколько рецензентов согласны. Тогда - через Guardian.

## 10. Что не работает автоматически

Несмотря на полноту схемы - есть **три места** где враждебная проверка сейчас не принуждается автоматически. Это **сознательные пробелы** (см. CLAUDE.md раздел Sprint AA «Social-discipline boundaries»). Их **нельзя закрыть разборщиком** потому что сигнал подделки идентичен сигналу законной короткой работы.

### Пробел 1 - Brief NOTE (ранее PRD)

`brief-intake` создаёт Brief, но **обязательной проверки роли B на Brief нет**. Считается что PRD - это уже Brief после враждебной проверки. На практике если PRD стартует с плохого Brief, проблема всплывает только в проверке PRD.

**Что делать вручную**: после создания Brief - короткий просмотр глазами или быстрый запуск `artifact-reviewer` на сам Brief.

### Пробел 2 - содержание гипотез в FPF ADI (Gap G7 из Sprint AA)

`forgeplan_reason` порождает минимум 3 гипотезы, но **выбор одной и качество её обоснования не проверяется автоматически**. Sprint Z7 (PRD-059) проверяет только структуру: «есть минимум 3 раздела `### Hypothesis`?». Содержание - нет.

**Почему не разбирается разборщиком**: 3 заглушки с пустым телом не отличить разборщиком от 3 коротких но настоящих гипотез. Это решается социальной дисциплиной - рецензент роли B должен проверить вручную.

### Пробел 3 - заглушка в `## Findings` (Gap G6 из Sprint AA)

Аналогично - EVID с телом `## Findings\n1. nothing wrong` пройдёт ворота guardian (Шаг 4b считает минимум 1 элемент). Sprint Z6 требует минимум 2 предложения обоснования при заявлении о нулевых пробелах, но автоматического разборщика на «минимум 2 предложения» нет.

**Почему не разбирается**: короткая законная находка типа «AC-3 не измеримо, нужен численный порог» - это 2 предложения и реально полезная находка. Отличить её от заглушки разборщиком нельзя.

### Пробел 4 - между волнами в `/sprint` (организационный, не разборщика)

В параллельных волнах через TeamCreate EVID собираются **на закрытии всей волны**, не после каждого агента. Если 5 кодеров отработали и 1 написал плохой код - враждебная проверка увидит это только когда все 5 закончили.

**Что делать**: для критичных волн - запускай рецензента **внутри волны** на промежуточное слияние, не жди закрытия.

### Общий образец - когда НЕ автоматизировать

Все четыре пробела имеют общую структуру:

1. Подделка ловится структурно (есть/нет раздел, количество элементов)
2. Но сигнал подделки идентичен сигналу законной короткой работы
3. Разборщик-ворота либо ложно срабатывает на законной работе (подрывает доверие), либо настолько мягкий что подделка проходит
4. Социальная дисциплина - видимый идентификатор рецензента, взаимная проверка, распознавание образца со временем - правильный слой защиты

Это обобщается: **не пиши разборщиков когда сигнал семантический, а не структурный**. Доверяй цепочке рецензентов. Делай их идентичность видимой.

## 11. Smith - выбор методологии под задачу

Когда непонятно с какой методологии начать - `/smith`. Это мастер-оркестратор который читает состояние проекта и возвращает план: какую методологию применить + последовательность запуска.

### Двенадцать контекстов матрицы

```
+----+-------------------------------+--------------------------------------+
| N  | Контекст                      | Основная методология                 |
+----+-------------------------------+--------------------------------------+
|  1 | Пустой репозиторий            | BMAD (упрощённый) + GitHub Spec Kit  |
|  2 | Модернизация legacy           | Strangler Fig + DDD + ACL            |
|  3 | Новая возможность в сервисе   | SPARC + Hexagonal Architecture       |
|  4 | Прод-ошибка (нетривиально)    | RIPER-5 + 5 Whys                     |
|  5 | Тривиальный хотфикс           | Без формальной методологии           |
|  6 | Рефакторинг                   | Branch-by-Abstraction + Mikado       |
|  7 | Архитектурное решение         | FPF ADI + ADR / MADR 3.0             |
|  8 | Аудит безопасности            | OWASP Top 10 2025 + STRIDE / ASTRIDE |
|  9 | Аудит производительности      | DORA + SRE error-budget + perf budget|
| 10 | Открытие продукта (PDLC)      | Jobs-To-Be-Done + Lean Startup       |
|    |                               | + Double Diamond                     |
| 11 | Чистка техдолга               | A3 + Fishbone + ADR-supersede        |
| 12 | Прод-инцидент в реальном врем.| Incident Command System              |
|    |                               | + разбор без поиска виноватых        |
+----+-------------------------------+--------------------------------------+
```

### Жёсткие правила smith

1. **Выбирается ровно одна строка**. Смешивать методологии запрещено - это даёт артефакты которые не вписываются ни в одну и заставляют команду изобретать чек-листы с нуля.

2. **Если ситуация на границе двух строк** - выбирается строка с **более высоким риском** (legacy > пустой, аудит > возможность). Отклонение фиксируется в разделе «Notes» плана.

3. **smith не пишет код, не активирует артефакты**. Только возвращает план - оркестратор решает что делать.

4. **smith ничего не запускает сам**. Пишет «запускаю X» и возвращает план - запуск происходит в основной сессии после возврата smith.

### Когда вызывать `/smith`

- В начале сессии когда непонятно что делать дальше
- На пустом репозитории (автоматически уходит в `/smith-bootstrap`)
- Для задачи нетривиального масштаба
- Когда `/forge-cycle` или `/autorun` не подходят (например - поток исправления ошибки с RIPER-5)
- Перед запуском многоспринтового EPIC

### Когда НЕ вызывать

- Тривиальные исправления одной строки - нагрузка процесса превышает стоимость исправления
- Исполнение известного запуска - если уже знаешь какого агента вызывать, вызывай напрямую
- Активация артефакта - это работа оркестратора + guardian

### Что возвращает smith

Markdown-план с **8 обязательными разделами**:

```
1. Тип контекста - одна из 12 строк
2. Основная методология + дополнительная
3. Последовательность запуска - нумерованный список агентов
4. Требования к доказательствам - чек-лист для ворот guardian
5. Риски + обратимость - что вынудит сменить маршрут
6. Заметки - если выбор на границе двух строк
7. Передача дел - для следующей сессии если работа на несколько дней
8. (плюс контекстные заметки от smith)
```

## 12. Словарь и сокращения

### Артефакты

- **PRD** (Product Requirements Document) - продуктовый документ требований, основной артефакт фазы Specification. 13 обязательных разделов.
- **RFC** (Request for Comments) - техническое предложение реализации, дочерний к PRD.
- **ADR** (Architecture Decision Record) - запись архитектурного решения в формате MADR 3.0.
- **EVID** (Evidence) - доказательный артефакт, выход рецензента роли B. Содержит вердикт и `## Findings`.
- **NOTE** - заметка свободного формата (Brief, трекер отложенных дел, мнения).
- **EPIC** - большая многоспринтовая инициатива, разбивается в N PRD.
- **PROBLEM / SOLUTION** - пара артефактов «проблема и решение», используется когда решение неочевидно.
- **SPEC** - узкая техническая спецификация (формат, протокол, схема).
- **REFRESH** - обновление устаревшего знания без полной замены.

### Методологии

- **BMAD** (Brainstorming -> Modeling -> Architecting -> Delivery) - четырёхфазная методология создания PRD.
- **FPF** (First Principles Framework) - каркас структурированного рассуждения. Включает ADI как основной цикл.
- **ADI** (Abduction -> Deduction -> Induction) - цикл рассуждения от Чарльза Пирса. Минимум 3 гипотезы.
- **SPARC** (Specification -> Pseudocode -> Architecture -> Refinement -> Completion) - пятифазная методология реализации возможности.
- **RIPER-5** (Research -> Innovate -> Plan -> Execute -> Review) - пятифазная методология исправления прод-ошибки.
- **Strangler Fig** - образец постепенной замены legacy-кода, по аналогии с фикусом-душителем.
- **DDD** (Domain-Driven Design) - методология моделирования предметной области с ограниченными контекстами.
- **ACL** (Anti-Corruption Layer) - защитная прослойка между новым и старым кодом.
- **OWASP** (Open Web Application Security Project) - стандарт классификации веб-уязвимостей.
- **STRIDE** - модель угроз от Microsoft (Spoofing/Tampering/Repudiation/InfoDisclosure/DoS/ElevationOfPrivilege).
- **MADR** (Markdown Architecture Decision Records) - формат записи ADR в Markdown.

### Слои конвейера

- **S10 FPF** - слой проектирования, требует ADI с минимум 3 гипотезами.
- **S11 BMAD** - слой ворот качества, требует проверки роли B с минимум 1 находкой.
- **S12 OpenSpec** - слой структуры, требует дельты при замене.
- **S13 Forgeplan** - слой автоматизации, требует правильной обвязки обработчиков/агентов/навыков/MCP.

### Роли агентов (CRUD-R-A)

- **Роль A - Создатель** - создаёт новые артефакты через MCP.
- **Роль B - Рецензент** - проверяет, создаёт EVID.
- **Роль B-оркестратор - Стратегический планировщик** (smith) - планирует, не исполняет.
- **Роль B-ворота - Финальный мастер** (guardian) - финальный вердикт.
- **Роль C - Только-чтение** - собирает контекст без сохранения.
- **Роль C-кодер** - единственный с правом писать код.
- **Роль D - Сопроводитель** - правит существующие артефакты на месте.

### Прочее

- **MCP** (Model Context Protocol) - протокол вызова инструментов forgeplan/Hindsight из помощника.
- **Обработчик** (hook) - bash-скрипт, реагирующий на события (PreToolUse, PostToolUse, SessionStart и так далее).
- **Заголовочный блок** (frontmatter) - YAML-блок в начале файла агента/навыка, определяющий метаданные (name, description, denylist, ...).
- **Список запретов** (denylist) - список запрещённых инструментов в заголовочном блоке. Физическая защита от нарушения роли.
- **Маркер** (sentinel) - служебная строка типа `<<NEED_USER_INPUT>>` или `<<NEEDS_ACTIVATION>>`, которую разбирает оркестратор.
- **R_eff** - эффективная надёжность, оценка 0..1, рассчитывается по формуле F+G+R.
- **Рабочее дерево** (worktree) - изолированная git-ветка для параллельной работы агентов роли C-кодер.

## Приложение Б. Памятка-чек-лист перед активацией Standard+ артефакта

Перед каждым `forgeplan_activate` для Standard+ артефакта пройти:

```
- /methodology-check <ID> - покрытие 100% по всем применимым слоям
- S10 FPF: EVID с минимум 3 разделами ### Hypothesis есть и связан
- S11 BMAD: минимум 1 EVID роли B с непустым ## Findings связан
- S12 OpenSpec: если замена - дельта заполнена (ADDED/MODIFIED/REMOVED/UNCHANGED)
- S12 OpenSpec: если 3+ модулей - диаграммы C4 L1+L2 созданы
- guardian (B-ворота) дал PASS
- R_eff артефакта приемлемый (обычно от 0.6 для Standard, от 0.8 для Critical)
- Никаких CONCERNS открытых на родительских артефактах
```

Если хотя бы один пункт не выполнен - **не активировать**. Возврат в конвейер на нужный шаг.
